## 1. 本次实现内容

已经实现报告中建议的 **Candidate-Constrained GRPO, CC-GRPO**。核心变化是把 GUI grounding 从“自由文本生成 JSON”改成“在当前 candidate id 集合内做离散动作选择”。

新增代码：

- `rl/candidate_constrained.py`
  - `reward_v3`：只奖励 candidate 几何质量，不再奖励 JSON 格式。
  - `candidate_completion()`：把 candidate id 固定成短 JSON completion。
  - `compute_candidate_completion_logprobs()`：teacher-forced 计算每个 candidate completion 的 logprob。
  - hard-negative/action-set 构建工具。
- `scripts/build_gui_candidate_preference_data.py`
  - 从 candidate RL JSONL 生成 `candidate_rewards_v3`、`cc_action_ids`、hard negatives、rank bucket。
- `scripts/train_gui_candidate_cc_grpo.py`
  - 自定义 CC-GRPO 训练循环。
  - 支持 `torchrun` 双卡 DDP。
- `scripts/eval_gui_candidate_scores.py`
  - 不生成文本，直接给 candidate id 打分并选择最高分 candidate。
  - 支持分块打分，避免全候选评估 OOM。

## 2. 算法层面改进

旧 GRPO：

```text
prompt + image -> 自由生成 '{"candidate_id":"cXX"}' -> parse -> reward
```

新 CC-GRPO：

```text
prompt + image + candidate list
-> 对合法 candidate completion 计算 logprob
-> 在有限 action set 内 softmax
-> 采样 K 个 candidate id
-> 用 reward_v3 计算组内 advantage
-> 更新 LoRA
```

关键点：

- 动作空间被限制为合法 candidate id，去掉 JSON 格式噪声。
- reward 只看 `pointing / IoU@0.5 / IoU shaped / center distance / top1 wrong penalty`。
- 每条样本构造 `oracle + hard negatives + random negatives`，主动制造组内 reward 方差。
- 训练中使用 `g4`，即每个 prompt 采样 4 个 candidate action。
- 训练 action set 为 8 个候选，记作 `a8`。

## 4. 双卡分布式训练方法

本次使用的是 **PyTorch DDP + torchrun + NCCL**，不是模型并行，也不是把两张 3090 拼成单张 48GB 显存。

启动方式：

```bash
PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True CUDA_VISIBLE_DEVICES=0,1 \
torchrun --standalone --nproc_per_node=2 scripts/train_gui_candidate_cc_grpo.py \
  --model /root/models/Qwen2.5-VL-3B-Instruct \
  --adapter checkpoints/qwen25vl_3b_gui_candidate_c60_sft_warmup_lora \
  --image-max-pixels 262144 \
  --train-data outputs/gui_candidate_rl_os_atlas_linux_2k_c60_cc_pref/train.jsonl \
  --output-dir outputs/gui_candidate_rl_os_atlas_linux_2k_c60_cc_pref/cc_grpo_ddp_g4_a8_100step \
  --max-steps 100 \
  --per-device-batch-size 1 \
  --num-generations 4 \
  --max-actions 8 \
  --max-hard-negatives 6 \
  --learning-rate 2e-6 \
  --policy-temperature 1.2 \
  --entropy-coef 0.01 \
  --logging-steps 5 \
  --gradient-checkpointing \
  --no-ddp-find-unused-parameters
```

实际分布式配置：

| 项 | 值 |
|---|---:|
| GPU | 2 x RTX 3090 |
| Method | `torchrun` DDP |
| Backend | NCCL |
| World size | 2 |
| 每卡 batch | 1 |
| 全局 batch | 2 |
| 可训练参数 | 29,933,568 LoRA params |
| Base parameters | 3,784,556,544 |

重要说明：

- DDP 是每张卡一份完整模型副本，然后同步梯度。
- 所以单卡 24GB 限制仍然存在；双卡主要提升吞吐和 batch，并不能把单个 forward 的显存变成 48GB。
- 为了让训练在 3090 上稳定，做了三处显存优化：
  - 使用 `logits_to_keep`，只保留 completion token 的 logits。
  - 设置 `image_max_pixels=262144`，和 SFT warmup 保持一致。
  - 使用 non-reentrant gradient checkpointing，避免 PEFT LoRA + DDP 的 ready-twice 问题。

## 6. 评估结果

### 6.1 cc_action_ids 口径，val100

这个口径使用 preference 数据里的 `cc_action_ids`，即 oracle + hard negatives 的候选子集。它适合验证“候选约束算法是否学会在高信号候选组里排序”，但不是最终真实部署指标。

| Adapter | Avg reward | Pointing | IoU@0.5 | Avg IoU | Avg rank | c00 rate |
|---|---:|---:|---:|---:|---:|---:|
| SFT warmup | 0.2552 | 11.00% | 9.00% | 0.0807 | 2.26 | 70.00% |
| CC-GRPO g4/a8/100 | 0.3457 | 20.00% | 17.00% | 0.1379 | 13.66 | 16.00% |

结论：在候选约束训练口径下，CC-GRPO 明显提升，并显著降低 `c00` 偏置。

### 6.2 全 candidate list 口径，val30

这个口径更接近真实部署：不使用 `cc_action_ids`，直接对完整 candidate list 打分。由于全候选打分很慢，本次先跑 val30。

| Adapter | Avg reward | Pointing | IoU@0.5 | Avg IoU | Avg rank | c00 rate |
|---|---:|---:|---:|---:|---:|---:|
| SFT warmup | 0.2529 | 10.00% | 6.67% | 0.0689 | 2.50 | 80.00% |
| CC-GRPO g4/a8/100 | 0.2958 | 13.33% | 6.67% | 0.0747 | 16.10 | 3.33% |

结论：全候选小样本下，CC-GRPO 也降低了 `c00` 偏置并提高了 avg reward / pointing，但 IoU@0.5 暂时持平。它开始愿意选中后排候选，这是进步，也是风险：还需要 listwise warmup 或更强 KL/正样本监督，避免过度远离高质量前排候选。

## 7. 当前结论

本次算法改进是有效的工程原型：

- 双卡 DDP 训练已跑通。
- CC-GRPO 不再受 JSON 格式噪声影响。
- hard-negative action set 显著改善 reward 方差。
- 100-step 后候选排序指标有正向变化。
- `c00` shortcut 被明显打破。

但这还不是最终 RL 方案：

- `cc_action_ids` 评估不是部署指标，只能证明候选组内排序能力。
- 全候选评估仍然慢，需要批量缓存视觉特征或两阶段 rerank。
- 全候选 val30 的 IoU@0.5 未提升，说明还需要加强候选质量和监督 warmup。

## 8. 下一步建议

1. 加一个 candidate-level listwise supervised warmup：

   ```text
   L = CE(softmax(candidate_rewards_v3 / tau), pi_theta(candidate | prompt))
   ```

   目标是先把 oracle / high-reward candidate 的概率拉上去，再做 CC-GRPO。

2. 把推理改成两阶段：

   ```text
   full candidates -> cheap query-aware shortlist top 12/16 -> CC scorer rerank
   ```

   这样避免全候选逐个 VLM scoring 太慢。

3. 跑 `CC-GRPO 300-step`，但建议加：

   - lower entropy coef 或 entropy schedule
   - KL/reference penalty
   - rank-balanced sampler
   - oracle-positive forced sampling

4. 扩大数据：

   当前仍是 `2k rows / 100 images` debug 规模。正式结果需要 `10k+ rows / 1k+ unique images`。
