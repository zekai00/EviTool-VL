## 项目一句话

构建了一个面向视觉语言模型的 GUI agentic RL 训练与评测框架，使 Qwen2.5-VL 能够看浏览器截图、输出结构化 GUI 动作，并在可执行环境中完成多步网页任务；通过 history-aware trajectory SFT、verifier-guided on-policy GRPO 和 replay 稳定训练，将正式 `val200` 成功率从 0.745 提升到 0.850，并实现了支持流式落盘和中断恢复的 on-policy 数据采集器。

## 面试讲述版本

这个项目的目标是让视觉语言模型不只是回答图片问题，而是像 GUI agent 一样看截图、理解目标、连续执行操作。早期我尝试过候选框选择式 GUI RL，也就是先生成一批可点击候选框，再让模型从里面选目标。但这条路线受候选池召回限制，容易变成选框排序任务，不是真正的多步 agent。

后来我把主线调整为可执行环境中的 VLM agentic RL。具体来说，我用 Playwright 搭了一个本地浏览器环境，统一 action JSON，包括 click、type、press、scroll、finish；每个任务都有 reset、step、screenshot、verifier 和 reward。这样模型每一步都能看到真实截图，输出动作，然后环境执行并返回结果。

数据上，我先构建 1000 个任务做 history-aware trajectory SFT。history-aware 的意思是每一步输入里包含当前截图、任务目标、历史动作、上一步 verifier progress 和动作空间，让模型学习状态到动作，而不是单张图到答案。通过坐标扰动和失败恢复示范，Qwen2.5-VL-3B 在独立 test100 上达到 0.82 success_rate，JSON/action 合法率都是 1.0。

随后我构建了 2000 任务正式套件，train/val/test=1600/200/200，并把 v3 SFT 模型在 val200 上的 0.745 作为正式 baseline。第一轮 on-policy GRPO 使用 100 个 trainable groups，把 val200 提升到 0.790。第二轮加入弱项 template 采样和 SFT replay loss，使用 213 个 trainable groups，把 val200 进一步提升到 0.850。

这说明 on-policy RL 链路是有效的：当前模型能自己采样动作，环境能执行，verifier 能给 reward，GRPO 能更新 LoRA，val gate 能检测提升和退化。不过当前还不是最终模型，因为 table 从 0.88 降到 0.76，advanced 仍为 0.6667，所以我保留了 test200，没有用它调参或验收。

最近我还修了采集器工程问题。之前 collect-only 只有进程结束后才写 groups，一旦长采集中断会损失 partial groups，也无法实时看到各类 trainable group 数量。现在采集器会实时写 `groups.jsonl.tmp`、`rollouts.jsonl.tmp` 和 `live_summary.json`，并支持 `--resume-collect` 从临时文件恢复；如果恢复后目标已满足，会直接退出，不再加载本地 Qwen。

## 面试高频问题回答

### 1. 这个项目和普通 VLM SFT 有什么不同？

普通 VLM SFT 多数是图像到文本答案；这个项目训练的是状态到动作。输入包含截图、任务目标、历史动作和 verifier progress，输出是可执行 GUI action JSON。评测也不是文本匹配，而是真实浏览器环境中的任务成功率。

### 2. 为什么不继续做候选框选择？

候选框路线受候选池召回限制。如果正确目标没有被生成，模型无法选中。更重要的是，它会把问题变成候选框排序，而目标是让 VLM 学会多步 GUI 操作。因此候选框可以作为辅助，但主线应是 direct GUI action 和可执行环境。

### 3. 为什么先做 SFT，再做 RL？

直接 RL 的搜索空间太大，模型一开始可能连合法 JSON action 都输出不稳定。SFT warmup 先让模型学会动作格式、坐标规则和基础流程。之后再做 on-policy GRPO，让模型从自己的成功和失败中学习。

### 4. 现在是否已经是真正 on-policy RL？

是。当前模型自己采样动作，环境执行动作，verifier 给 reward，GRPO 用这些当前策略产生的数据更新 LoRA。第一轮 100 groups 把 val200 提到 0.790，第二轮 213 groups replay 版把 val200 提到 0.850。

### 5. 为什么 0.850 还不跑 test200？

因为总体分数高，但 table 和 advanced 有明显退化。table 从 v3 的 0.88 降到 0.76，advanced 从 0.7778 降到 0.6667。test200 应保留给总体高且主要 family 不明显退化的版本，不能拿来反复调参。

### 6. 采集器改造解决了什么问题？

之前 collect-only 结束后才写 groups，长采集中断会丢 partial data。现在每采到 group/rollout 就写临时文件，并实时更新 live summary；中断后用 `--resume-collect` 可以继续，且目标已满足时不会重新加载模型。

### 7. 下一步怎么做？

下一步专门采 table_action 和 advanced_scroll/dialog，用 replay 或更高 loss weight 修复这两个 family。目标是保持 val200 >= 0.85，同时 table >= 0.84、advanced >= 0.7778。达标后再跑 test200，之后再迁移到 Qwen2.5-VL-7B。
