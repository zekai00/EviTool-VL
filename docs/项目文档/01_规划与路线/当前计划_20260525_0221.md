## 2. 当前存在的问题

### 2.1 3B Tool 模式学会了“直接答”，而不是稳定用工具

现象：

- 3B post-SFT tool 的 `avg_tool_calls` 从 1.45 降到 0.34。
- 大量样本第一步直接输出 final answer。
- Text relaxed 上升，但 evidence closed 只从 18% 升到 30%，仍很低。
- GUI pointing 从 56% 掉到 28%。

可能原因：

1. 训练集中加入了 direct retention 样本。
   - 这对保留 Text QA 有帮助。
   - 但对 3B 这种协议能力弱的模型，会强化“可以直接答”的行为。
2. 训练数据的 user prompt 写了 “Use local visual tools when evidence is needed”。
   - 这句话允许模型判断“不需要 evidence”，从而直接 final。
   - 与 `eval_tool_baseline` 默认要求首步 action 的协议不完全一致。
3. SFT 数据同时混合 direct 和 tool trace。
   - 3B 可能没有学会区分 direct 模式和 tool 模式。
   - 结果是在 tool eval 中也套用了 direct-answer 行为。
4. 3B 原始协议错误率很高。
   - pre-SFT protocol error 是 73%。
   - 第一轮 SFT 样本量只有 867，对协议修复仍不足。

解决方案：

- 单独训练 3B tool-only adapter。
- 去掉 direct retention。
- 修改训练 prompt：
  - 首轮必须输出 ACTION JSON。
  - 不允许第一步 FINAL JSON。
  - FINAL 必须引用 evidence id。
- 训练数据中加入更多“错误协议修复样本”：
  - 缺 evidence 的 final 改写。
  - `bbox_2d` 改成 `bbox`。
  - action/final 混写拆分。
- 评测 3B 当前 adapter 时先加 `--force-tool-first`，确认退化是否主要来自“第一步直接 final”。

### 2.2 4B Tool SFT 损伤了原本很好的 tool 能力

现象：

- 4B pre-SFT tool 已经很强：
  - Protocol error：0%
  - Evidence closed：100%
  - Final parse：100%
- post-SFT 后：
  - Text relaxed：77.33% -> 68.00%
  - Evidence closed：100% -> 96%
  - Final parse：100% -> 96%
- 只有 GUI IoU@0.5 小涨：
  - 12% -> 16%

可能原因：

1. 4B 原本 prompt-only tool 协议已经很好。
   - SFT 对协议收益有限，反而扰动原有能力。
2. 训练数据质量不均衡。
   - ChartQA strong evidence rate 只有 46%。
   - AI2D strong evidence rate 只有 16%。
   - 这些样本会教模型形成“粗 evidence 也能答”的行为。
3. 训练目标过混合。
   - 同时训练 GUI、DocVQA、ChartQA、AI2D、direct retention。
   - 对 4B 来说，可能不如保留原能力，只做小规模 targeted SFT。
4. 学习率可能偏高。
   - `1e-4` 对已有协议能力较强的 4B 可能过强。
   - 造成 Text QA 和 evidence 行为退化。

解决方案：

- 4B 不做全任务 SFT。
- 改成小学习率 targeted SFT：
  - lr 降到 `5e-5` 或 `3e-5`。
  - 只训 GUI + 强证据样本。
  - 不混入弱 ChartQA/AI2D。
- 或者保留 4B pre-SFT prompt-only tool 作为当前 tool 主模型。
- 4B post-SFT adapter 暂时只作为 direct/text 对比参考，不作为主推理模型。

### 2.3 GUI Detect 仍未达到论文理想目标

当前结果：

| 指标 | 当前 | 理想目标 |
|---|---:|---:|
| Recall@30 | 80.20% | 85%+ |
| Oracle GT 注入比例 | 19.80% | <15% |

问题：

- 中期目标已经达成，但距离论文理想目标仍有差距。
- OCR fallback 使用率 37.80%，说明初始视觉候选还不够强。
- Candidate Recall@1/3/5 偏低：
  - Recall@1：19.20%
  - Recall@3：31.60%
  - Recall@5：37.80%

可能原因：

1. 候选召回够了，但排序不够好。
   - Recall@30 已经 80.20%，但 Recall@5 只有 37.80%。
   - 说明目标经常在候选池里，但排得靠后。
2. 小图标、系统栏、浏览器控件仍难检测。
3. text button 与 icon button 的统一候选建模不够。
4. query-aware prior 仍是启发式规则，没有学习型 reranker。

解决方案：

- 训练或实现 query-aware candidate reranker。
- 将候选排序任务显式建模：
  - 输入：instruction + candidate list + OCR text + bbox geometry。
  - 输出：best candidate id。
- GUI trace 不只训练最终 bbox，而要训练：
  - detect -> candidate selection -> click/final。
- 增加 hard case mining：
  - 收集 `oracle_gt_injected=true` 的样本。
  - 分析 miss_reason。
  - 针对小图标、顶部栏、系统菜单做 detector/ranker 修复。

### 2.4 ChartQA evidence 仍弱

当前数据指标：

- Answer-in-OCR Rate：46.00%
- Answer Evidence Position Rate：46.00%
- Chart Structure Evidence Rate：100.00%
- Chart Structure Only Rate：54.00%

问题：

- 超过一半 ChartQA 样本只有结构 evidence，没有答案位置 evidence。
- 对数值型图表问题，OCR 不一定能直接读出答案。
- 当前 bar candidates 只是结构候选，不等于可验证数值 evidence。

可能原因：

1. ChartQA 很多答案需要计算或比较，不是图上直接 OCR 文本。
2. 当前 `detect(mode=bar)` 能找结构，但没有完整 axis/legend/value linking。
3. 没有把 bar height、axis ticks、legend、series name 组织成结构化证据。

解决方案：

- 增强 ChartQA 工具证据：
  - axis tick OCR。
  - bar/line/point candidate。
  - legend OCR。
  - bar height 到数值的估算。
- 增加 chart reasoning trace：
  - detect chart structure。
  - OCR ticks/labels。
  - measure bar heights。
  - final answer with evidence。
- 对 ChartQA 区分两类样本：
  - direct answer in OCR。
  - derived numeric answer。
- SFT 时不要把弱 ChartQA 当 strong evidence 使用。

### 2.5 AI2D strong evidence 仍低

当前数据指标：

- Strong evidence rate：16.00%

问题：

- AI2D 从 0 提升到了 16%，但仍是所有任务里最弱。
- 大多数样本只有 diagram structure evidence，没有正确选项位置 evidence。
- 训练模型可能只学会“看过图”，没有学会“证据指向正确答案”。

可能原因：

1. AI2D 的答案通常是选项文本或概念，不一定直接出现在图中。
2. OCR 对图示 label 的召回有限。
3. 当前没有 label-object linking。
4. 没有把箭头、部件、标签、选项之间的关系显式建模。

解决方案：

- 增加 label-object linking：
  - OCR label bbox。
  - diagram object bbox。
  - arrow/line relation。
  - option text 与 diagram label 的匹配。
- 对 AI2D 样本分级：
  - strong：正确选项或图中标签可定位。
  - moderate：有图结构但无答案位置。
  - weak：结构也不可靠。
- SFT 强证据训练优先 strong。
- moderate 样本只用于工具协议训练，不用于答案正确性强监督。

## 4. 当前推荐策略

### 4.1 当前最佳模型使用方式

短期不要把第一轮 SFT adapter 作为最终主模型。

推荐：

- 3B：
  - direct 场景可用 post-SFT adapter。
  - tool 场景暂不推荐使用 post-SFT adapter，先做 force-tool-first 验证。
- 4B：
  - tool 场景继续使用 pre-SFT prompt-only baseline。
  - post-SFT adapter 只作为 direct text 改善参考。
- detect/ui：
  - 当前已达中期目标，可以进入 reranker 阶段。
- AI2D：
  - 当前增强有效，但 strong evidence 仍弱，需要继续做 label-object linking。

### 4.2 不建议现在做的事情

暂时不建议：

- 直接上 GRPO。
- 继续用当前混合数据全任务 SFT。
- 用弱 evidence ChartQA/AI2D 样本强化答案训练。
- 把当前 4B post-SFT tool 当主结果。
- 只看 Text relaxed，不看 evidence closed 和 avg tools。

原因：

- 当前最大问题不是模型不会答，而是工具调用策略和证据闭环没有稳定。
- 如果现在做 RL，reward 可能会放大“少调用工具直接答”的行为。
- 必须先把 SFT 数据和协议行为修正，再做 RL。

  - trace/data mixture
- detect/reranker 和 evidence labeling 是提升上限的关键。
- RL 应该作为 SFT 后的第二阶段，而不是当前立即使用。
