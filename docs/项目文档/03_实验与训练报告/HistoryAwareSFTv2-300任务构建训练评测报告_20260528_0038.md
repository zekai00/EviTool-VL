## 术语说明

- `SFT`：监督微调，用已经知道正确答案的样本训练模型。本轮的答案是下一步 GUI action JSON。
- `History-aware`：带历史状态的提示词。模型输入里包含当前 step、最大 step、上一轮 verifier progress、最近动作历史、动作空间和截图。
- `Rollout`：一次完整任务尝试，从 reset 后的第一张截图开始，反复输出动作、执行、拿新截图和 reward，直到成功或到达步数上限。
- `Verifier`：验证器，用页面内部只读状态检查任务是否完成，并返回 progress 和 reward。
- `Oracle`：脚本专家，这里用 Playwright selector 精确点击或输入，用来生成高质量示范轨迹。
- `Policy`：策略，也就是决定下一步动作的主体。本报告里有 scripted oracle policy 和 local Qwen policy 两种。

## 失败模式

主要失败不是 JSON 格式，而是坐标/布局泛化。

例子 1：`suite_form_055`

- oracle 第一步点击 name 输入框：`{"action":"click","x":426,"y":381}`
- local_qwen 第一步输出：`{"action":"click","x":432,"y":300}`
- 结果：点击落在错误高度，后续 type 没有写入 name，verifier progress 一直是 `name_value=false, code_value=false`。

例子 2：`suite_search_055`

- oracle 搜索框点击：`x=352, y=289`
- local_qwen 搜索框点击：`x=343, y=231`
- 结果：type 后没有触发有效搜索，后续连续点击结果区域也无法成功。

例子 3：`suite_table_047`

- oracle 目标行点击：`x=621, y=735`
- local_qwen 反复点击：`x=625, y=691`
- 结果：点击行偏上，任务失败。

这说明当前 300 条任务还不足以让 3B 级 VLM 在多布局网页中稳定学会精确坐标。模型能生成合法动作，但视觉 grounding 还弱。

## 当前判断

这轮完成了真正 on-policy VLM agentic RL 前的必要 warmup 数据和本地模型闭环，但还不建议直接上大规模 RL。

原因：

1. current model 的 test success_rate 只有 0.333。
2. 失败主要来自坐标 grounding，RL 会在低成功率下产生大量无效轨迹，样本效率差。
3. 300 任务覆盖仍偏少，每类 train 任务数量实际只有几十个。

下一步更稳妥的顺序：

1. 扩到 1000-2000 个 local browser tasks，重点增加 form/search/menu/table 的布局变化。
2. 加入坐标扰动增强：对同一目标元素生成多个落在元素内部的可执行点击点，让模型学到“点在区域内”而不是记一个中心点。
3. 增加失败恢复示范：例如 click 错后根据 verifier progress 重新定位，不只训练理想 oracle。
4. 训练 2-3 epoch，并加入 val/test 早停评测。
5. test success_rate 达到 0.6 左右后，再开始小规模 on-policy GRPO 或 verifier-guided RL。
