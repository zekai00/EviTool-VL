## 2. 当前 candidate RL 到底解决了什么

当前 GUI candidate RL 训练的是一个较明确的能力：

> 给定 GUI 截图、用户 instruction 和一组候选框，模型选择正确 `candidate_id`。

它能作为 tool-agent 的最后一步 grounding head，例如：

```json
{"candidate_id": "c114"}
```

系统再把 `candidate_id` 映射成 bbox 或 click point。

这个能力重要，但它不是完整的 VLM agent。它默认候选框已经存在，模型并不需要自己决定是否 OCR、是否裁剪、是否放大、是否检测、是否继续探索。

## 3. 为什么 candidate RL 不应继续作为第一优先级

按第一性原理，目标如果是 evidence-closed visual tool agent，核心问题不是“候选框已经给好后怎么选”，而是：

> 模型如何主动获取视觉证据，并基于工具返回的 observation 做下一步决策？

candidate RL 的局限：

- action space 是外部候选池给定的。
- 候选框缺失时，RL 无法从不存在的 action 中学会选对。
- 继续堆规则会越来越数据集特化，泛化收益下降。
- 它主要优化最后一步 selection，不训练主动工具使用策略。

当前 refined 候选池已经足够支撑下一阶段，不建议继续把大量时间花在手工规则上。

## 5. 下一阶段主线：Tool-Call Trajectory 训练

下一阶段应转向 VLM agentic RL，即训练模型完成多步工具调用轨迹：

```text
observe screenshot
-> call detect/ocr/crop/zoom
-> read tool observation
-> decide next tool or final answer
-> finish with bbox/click/answer
```

建议 action schema：

- `ocr(region | bbox | full_image)`
- `detect(mode, query, region)`
- `crop(bbox)`
- `zoom(bbox)`
- `measure(bbox | object)`
- `select_candidate(candidate_id)`
- `answer_bbox(bbox)`
- `click(point | candidate_id)`
- `finish(answer)`

其中 `select_candidate` 复用当前 candidate RL 结果；`answer_bbox` 支持无候选框时直接输出坐标。

## 6. 建议的实现路线

第一步：定义统一 trajectory schema。

每条样本包含：

- `image`
- `task`
- `messages`
- `tool_calls`
- `observations`
- `final_answer`
- `reward_signals`
- `metadata`

第二步：自动构造 oracle / silver trajectories。

可从现有数据生成几类模板：

- `detect_ui -> select_candidate`
- `ocr_full -> detect_query -> select_candidate`
- `crop_menu_region -> ocr -> select_candidate`
- `detect_grid -> answer_bbox`
- `crop_target_area -> answer_bbox`

第三步：做 trajectory SFT。

目标是先让模型学会：

- 合法 JSON 工具调用格式。
- 什么时候调用哪个工具。
- 如何读取 observation。
- 如何停止并输出最终答案。

第四步：做 agentic RL。

reward 设计：

- final correctness：bbox/click/answer 是否正确。
- evidence closed：observation 是否包含目标文本/区域/对象。
- tool validity：工具调用格式和参数是否合法。
- efficiency：少调用无效工具。
- grounding quality：最终 bbox IoU / pointing / candidate reward。

第五步：做三种评测。

- direct mode：不给工具，直接输出 bbox/answer。
- candidate mode：给候选框，选择 candidate。
- tool-agent mode：模型自己调用工具完成任务。

## 7. 当前不建议继续投入的方向

暂时不建议继续大规模做：

- 更多 LibreOffice/GNOME hard-coded candidate prior。
- 为了单个 data_type 继续堆手工框规则。
- 在没有真实 current-model score cache 前反复训练 CC-GRPO。

这些工作会让当前 benchmark 指标更好，但不一定提升 VLM agentic RL 的核心能力。
