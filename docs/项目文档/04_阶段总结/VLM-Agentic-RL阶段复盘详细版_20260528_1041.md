## 关键技术名词

- `VLM`：Vision-Language Model，视觉语言模型。这里指能同时理解图像和文本的模型，例如 Qwen2.5-VL、Qwen3-VL。我们的目标不是只让它回答图片问题，而是让它看 GUI 截图后决定下一步操作。
- `GUI`：Graphical User Interface，图形用户界面。网页、桌面应用、表单、菜单、表格都属于 GUI 场景。
- `Agentic RL`：智能体强化学习。模型不只输出一个静态答案，而是在环境中多步观察、行动、获得奖励，并逐步学会完成任务。
- `Tool-call trajectory`：工具调用轨迹。指模型在每一步调用一个工具或动作，例如 `click`、`type`、`press`、`scroll`，多步连起来形成完整任务轨迹。
- `Candidate box`：候选框。早期路线中，我们先生成一批可能可点击的 GUI 区域框，再让模型从这些候选框中选一个。它能降低动作空间难度，但容易把问题变成“从候选框里选答案”，不是完整的 GUI agent。
- `Full-aware candidate`：让模型在知道完整候选框集合的情况下做选择或打分。它对候选框排序有用，但仍然依赖候选池质量。
- `Playwright`：浏览器自动化工具。它可以在程序里打开网页、点击、输入、截图。我们用它实现本地 browser GUI-RL 环境。
- `reset/step/obs/reward`：强化学习环境的基本接口。`reset` 是重置任务，`step` 是执行一步动作，`obs` 是观察到的截图和文本信息，`reward` 是当前动作得到的奖励。
- `Verifier`：验证器。用程序检查任务是否完成，例如表单是否填对、按钮是否点击、目标项是否选中，并返回成功状态和奖励。
- `Rollout`：一次完整任务尝试。从环境 reset 开始，模型连续执行多步动作，直到成功、失败或达到最大步数。
- `SFT`：Supervised Fine-Tuning，监督微调。用专家示范数据训练模型，让模型先学会合法动作格式和基础操作策略。
- `Trajectory SFT`：轨迹监督微调。训练样本不是单张图一个答案，而是多步轨迹中的每一步：给模型当前截图、历史动作、目标和状态，让它输出下一步 action。
- `History-aware`：带历史信息。提示词里包含当前 step、之前做过什么、上一步 verifier progress、动作空间和坐标规则，用来减少重复错误和状态误判。
- `Scripted oracle`：脚本专家策略。不是大模型，而是我们根据任务模板和 DOM 元素位置写出的正确操作生成器，用来生成高质量专家轨迹。
- `Coordinate jitter`：坐标扰动增强。点击目标不只点中心，而是在元素内部随机采样一些可执行坐标，提升模型对轻微坐标偏差的鲁棒性。
- `Failure recovery demo`：失败恢复示范。轨迹里故意加入一条 near-miss 失败动作，再演示如何根据历史和 verifier progress 恢复到正确动作。
- `LoRA`：Low-Rank Adaptation，低秩适配微调。只训练少量新增参数，减少显存和训练成本。
- `4bit quantization`：4 比特量化。用更低精度加载模型权重，降低显存占用。
- `Freeze vision tower`：冻结视觉塔。视觉编码器部分不训练，只训练语言侧和 LoRA 参数，减少训练成本和过拟合风险。
- `Current model`：当前策略模型。现阶段指本地训练后的 Qwen2.5-VL-3B LoRA，用它在环境中真实执行任务。
- `On-policy RL`：在线策略强化学习。当前模型自己采样 rollout，然后用这些当前策略产生的数据更新当前模型。我们还没有真正完成这一步，当前完成的是进入这一步之前的 SFT warmup。
- `GRPO`：Group Relative Policy Optimization，一类适合大模型 RL 的策略优化方法。核心思路是同一任务采样多条输出，用组内相对奖励估计优势，更新模型倾向于高奖励动作。
- `Verifier-guided RL`：验证器引导的强化学习。奖励来自 verifier，而不是人工标注或只看文本匹配。

## 阶段 1：候选框路线和它的局限

最早我们围绕 GUI candidate 做了一条候选框 RL 路线。基本思路是：先为每张 GUI 图生成很多可能的交互区域，也就是候选框；再让模型学习从候选框里挑选最接近目标的框。这条路线的优点是工程上可控，动作空间从任意坐标缩小为有限候选集合，适合做 reward shaping、candidate scoring、oracle hit、IoU 等指标。

这条路线也暴露了核心瓶颈：

1. 候选召回决定上限。正确目标如果不在候选池里，模型再强也选不到。
2. 很多精力会花在补框、候选框规则、候选池增强上，而不是让 VLM 真正学会多步操作。
3. candidate-only GRPO 容易退化为“选框排序模型”，和目标中的 VLM agentic tool-call trajectory 有距离。

因此我们后续明确：候选框、full-candidate score cache、candidate-only GRPO 不是当前第一优先级。它们可以作为辅助模块存在，但主线应转向真实可执行环境中的多步 GUI agent。

## 阶段 2：路线调整为真正的 VLM agentic RL

在多次讨论后，我们把目标收束为：让 Qwen2.5-VL/Qwen3-VL 这类 VLM 看到截图、理解任务目标、输出结构化 GUI action，并在环境中多步执行。

这意味着训练数据不应只是静态图片和 bbox，也不应只是候选框选择，而应该是：

- 当前截图
- 用户目标
- 当前 step
- 历史动作
- verifier progress
- 下一步 action JSON
- 执行后的 reward 和新截图

也就是说，数据单元从“图像到答案”变成“状态到动作”，项目形态从离线视觉定位转向可执行 agent 环境。

## 阶段 3：本地 browser GUI-RL 环境

我们没有直接上 OSWorld 这类复杂系统环境，而是先从 browser Playwright 环境做起。原因是浏览器任务更容易 reset、更容易用程序 verifier 验证、更适合快速迭代算法闭环。

当前已实现的环境能力包括：

- 用 Playwright 打开本地 HTML 任务页。
- 保存每一步截图。
- 接收统一 JSON action，例如 `click`、`type`、`press`、`scroll`、`finish`。
- 将 0-1000 归一化坐标映射到实际浏览器视口坐标。
- 执行动作后调用 verifier 检查任务进度和奖励。
- 记录完整 rollout，包括 task id、目标、动作、奖励、截图、执行状态。

这一步的意义很大：我们不再只做离线数据集，而是有了能 reset、能 step、能 verifier、能记录失败轨迹的本地 GUI-RL Gym。这里的 Gym 不是特指 OpenAI Gym 包，而是指一套强化学习环境接口。

## 当前模型到底能做什么

当前 Qwen2.5-VL-3B LoRA 能在我们构造的本地 browser GUI-RL 任务里：

1. 读取当前页面截图和任务目标。
2. 根据历史动作和 verifier progress 判断下一步。
3. 输出合法 JSON action，例如点击、输入、按键、滚动。
4. 在真实 Playwright 浏览器环境中执行多步任务。
5. 在 test100 上达到 0.82 success_rate。

它不是只能从候选框里选正确框。当前路线已经从 candidate-only 转为 direct GUI action，也就是模型直接输出坐标和动作类型。但它仍需要 Playwright 这类执行器来把动作落到真实环境里；这正是 visual tool agent 的形式。

## 仍然没有完成的部分

当前成果还不是完整的 on-policy VLM agentic RL。原因是：

1. v3 训练数据主要来自 scripted oracle，不是当前模型自己采样的数据。
2. 模型失败后的 rollout 还没有被用于策略梯度更新。
3. verifier reward 目前用于评测和轨迹记录，还没有真正进入 GRPO 更新。
4. 任务仍是本地 browser synthetic tasks，不是 OSWorld 级别复杂真实系统任务。

因此最准确的定位是：我们已经完成了 on-policy RL 前的可执行环境、SFT warmup、评测闭环和 current model 准备。

## 主要问题和下一步

当前主要瓶颈：

1. form 类任务成功率 0.6364，常见问题是输入框、submit 按钮、字段切换坐标不稳。
2. choice 类任务成功率 0.6667，常见问题是 select、checkbox、radio 等状态切换控件容易误点或恢复失败。
3. recovery 对状态切换控件需要单独处理，因为 near-miss 可能改变真实页面状态。
4. 当前任务还偏合成，需要逐步加入更接近真实网页的任务。

下一步建议：

1. 用当前 v3 LoRA 作为 current model，启动小规模 verifier-guided GRPO / on-policy RL。
2. 每个任务采样多条 rollout，用 verifier reward 给成功、部分进展、重复失败、无效动作打分。
3. 将失败轨迹、重复点击、reward 无进展状态纳入 RL 训练，而不是只依赖 oracle。
4. 同时补少量 form/choice 定向 SFT 修复数据，作为辅助 warmup。
5. 当 3B on-policy RL 闭环跑通后，再迁移到 Qwen2.5-VL-7B-Instruct。

## 阶段性判断

这个阶段最大的进展不是某个单点指标，而是项目主线已经从“离线候选框定位”切换到“真实可执行 GUI agent 训练”。我们现在有任务、有环境、有 verifier、有本地模型、有训练数据、有评测脚本、有失败分析，也有可以继续做 RL 的 current model。

接下来工作的核心不是继续无限补静态数据，而是让当前模型自己执行、自己失败、用 verifier 奖励学习恢复。这才是 VLM agentic RL 的主线。
