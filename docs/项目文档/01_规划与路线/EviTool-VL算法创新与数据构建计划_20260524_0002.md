# EviTool-VL 算法创新与数据构建计划

日期：2026-05-23

本文用于重新界定 EviTool-VL 的研究问题、算法创新点、数据规模、token 预算和实验路线。核心结论是：本项目不应被表述为“发明 GRPO”或“泛泛做视觉 CoT”，而应定位为 **面向小参数 VLM 的可验证视觉工具推理训练框架**。重点是让 3B-4B VLM 学会在图表、文档、科学图示和 GUI 任务中主动获取局部视觉证据，并让工具调用、证据引用、最终答案之间形成可检查闭环。

## 1. 结论先行

### 1.1 ScreenSpot 直接坐标全 0，不代表不能训练

百炼模型和本地 Qwen3-VL-4B 在 ScreenSpot direct setting 上 `IoU@0.5 = 0`、`Pointing = 0`，说明它们不擅长 **裸输出精确 bbox/坐标**。这不是训练不可行，而是训练目标设置不对。

GUI grounding 的真实瓶颈不是“模型完全看不懂按钮”，而是：

- 高分辨率截图中目标元素很小；
- 坐标是连续空间，语言模型用文本 token 直接回归坐标天然不稳定；
- VLM 视觉 token 粒度较粗，patch-level 表征与 pixel-level 坐标之间存在映射误差；
- 目标经常是图标、空白控件、相邻文本、小按钮，容易被附近视觉元素干扰；
- 只让模型直接输出 `[x1,y1,x2,y2]`，没有给它候选区域、局部 crop 或 evidence id。

因此训练策略应从：

```text
image + instruction -> bbox
```

改成：

```text
image + instruction
  -> detect(ui/text) 生成候选框
  -> crop/zoom 局部检查
  -> click(candidate_id 或 center)
  -> final answer 引用 evidence
```

也就是说，先把问题变成 **候选区域召回 + 候选选择 + 点击点输出**，再逐步要求高质量 bbox。对 GUI 来说，`Pointing` 比 `IoU@0.5` 更接近实际点击成功率；训练初期应优先优化 point-in-box。

### 1.2 现阶段不做 30K，先做 1K-2K 高质量 trace

公开数据集已经够用来做原始任务池：ChartQA、DocVQA、AI2D、ScreenSpot-v2。真正缺的不是基础 QA，而是本项目定义的工具轨迹：

```text
assistant action JSON -> local tool observation -> assistant final JSON
```

每条轨迹要有真实工具返回的 evidence id、OCR span、候选框、click point 或 bbox。公开数据集一般不会提供这些。

在每个模型只有 100 万 token 免费额度的限制下，不应盲目构建 30K。合理路线：

- v0：`1K-2K` 条高质量 tool traces；
- GUI 占比至少 50%，因为当前最大短板在 grounding；
- 每条 trace 只用 1 个 teacher 主模型生成候选计划，必要时少量二次修正；
- 本地工具和规则过滤承担主要质量控制，不做多模型大规模投票。

### 1.3 算法创新不在 GRPO 本身

GRPO 最早由 DeepSeekMath 系统性提出，用于降低 PPO 中 value model 的显存和训练成本。后续 R1-VL、MM-Eureka、GRPO-CARE、Video-VER、VG-CoT 等工作已经研究过多模态 RL、step-wise reward、grounded CoT、visual evidence reward。

因此，本项目不能声称：

- “我们首次提出 GRPO”；
- “首次提出每步引用 evidence”；
- “首次提出组合奖励”。

更准确的创新表述是：

> 面向小参数 VLM，构建一个本地可执行视觉工具环境，将工具调用、视觉证据、最终答案和反文本偏置鲁棒性统一到可验证 reward 中，用 SFT + RLVR/GRPO 训练模型从 prompt-only tool use 变成 evidence-closed visual tool agent。

## 2. 相关工作与现有方法的问题

### 2.1 GRPO 已是成熟基础算法

DeepSeekMath 提出 Group Relative Policy Optimization，目标是用 group 内 reward 均值和方差估计 advantage，从而省掉 PPO 的 value function。论文明确指出 GRPO 是 PPO 的变体，用于提升数学推理并优化 PPO 显存成本。

参考：

- DeepSeekMath: Pushing the Limits of Mathematical Reasoning in Open Language Models, arXiv:2402.03300  
  https://arxiv.org/abs/2402.03300

GRPO 的基本思想：

对同一个 prompt `x`，采样一组输出：

```math
\{y_i\}_{i=1}^{G} \sim \pi_\theta(\cdot|x)
```

用 reward function 得到：

```math
R_i = R(x,y_i)
```

组内归一化 advantage：

```math
A_i = \frac{R_i - \mu_R}{\sigma_R + \epsilon}
```

其中：

```math
\mu_R = \frac{1}{G}\sum_{j=1}^{G}R_j,\quad
\sigma_R = \sqrt{\frac{1}{G}\sum_{j=1}^{G}(R_j-\mu_R)^2}
```

然后使用 clipped policy objective 更新模型。GRPO 的优点是省显存、适合 verifiable reward；缺点是如果 reward 过稀疏、组内样本同错、或者 final-answer-only 奖励太强，模型仍然容易走捷径。

### 2.2 多模态 RL 已经在做 step-wise reward

R1-VL 提出 Step-wise GRPO，用规则奖励中间推理步骤；MM-Eureka 将 rule-based RL 用到多模态数学推理；GRPO-CARE 指出普通 answer-only GRPO 会提升答案但损害 reasoning-answer consistency，并提出 consistency-aware reward。

参考：

- R1-VL: Learning to Reason with Multimodal Large Language Models via Step-wise Group Relative Policy Optimization, arXiv:2503.12937  
  https://arxiv.org/abs/2503.12937
- MM-Eureka: Exploring the Frontiers of Multimodal Reasoning with Rule-based Reinforcement Learning, arXiv:2503.07365  
  https://arxiv.org/abs/2503.07365
- GRPO-CARE: Consistency-Aware Reinforcement Learning for Multimodal Reasoning, arXiv:2506.16141  
  https://arxiv.org/abs/2506.16141

这说明“组合奖励”和“步骤级奖励”不是空白点。本项目必须更具体：我们的 step reward 不是一般逻辑完整性，而是 **工具 observation 是否真实支持该步视觉推理**。

### 2.3 Grounded CoT 和 visual evidence 也已有相关工作

Argus、VLM-R3、VG-CoT、Video-VER 都在强调视觉证据、区域聚焦、grounded chain-of-thought 或 visual evidence reward。

参考：

- Argus: Vision-Centric Reasoning with Grounded Chain-of-Thought, arXiv:2505.23766  
  https://arxiv.org/abs/2505.23766
- VLM-R3: Region Recognition, Reasoning, and Refinement for Enhanced Multimodal Chain-of-Thought, arXiv:2505.16192  
  https://arxiv.org/abs/2505.16192
- When Thinking Drifts: Evidential Grounding for Robust Video Reasoning, arXiv:2510.06077  
  https://arxiv.org/abs/2510.06077
- VG-CoT: Towards Trustworthy Visual Reasoning via Grounded Chain-of-Thought, arXiv:2604.21396  
  https://arxiv.org/abs/2604.21396

这些工作证明该方向是合理且前沿的，但也压缩了“概念创新”的空间。本项目应避开笼统 grounded CoT，转向更工程可验证的研究点：

- 使用本地确定性工具，而不是只让模型在文本里声称“看到了”；
- 每个 evidence id 对应真实 tool observation；
- reward 能检查 answer、tool trace、evidence closure、OCR span、grounding、anti-text-bias；
- 目标模型是 3B-4B 小 VLM，强调低算力可复现训练闭环。

### 2.4 GUI grounding 的问题更接近坐标映射与小目标定位

ScreenSpot-Pro 指出，高分辨率专业 GUI 中目标小、环境复杂，现有 MLLM grounding 表现很弱；ScreenSeekeR 通过强 planner 引导级联搜索显著提升准确率。GUI-Actor 进一步指出，直接把 grounding 当成文本坐标生成会有空间语义对齐弱、坐标监督目标歧义、dense pixel 坐标与 coarse visual patch 不匹配等问题。RULER coordinate mapping 则说明显式坐标标记能改善高分辨率 GUI grounding。

参考：

- ScreenSpot-Pro: GUI Grounding for Professional High-Resolution Computer Use, arXiv:2504.07981  
  https://arxiv.org/abs/2504.07981
- GUI-Actor: Coordinate-Free Visual Grounding for GUI Agents, arXiv:2506.03143  
  https://arxiv.org/abs/2506.03143
- UI-AGILE: Advancing GUI Agents with Effective Reinforcement Learning and Precise Inference-Time Grounding, arXiv:2507.22025  
  https://arxiv.org/abs/2507.22025
- Improving GUI Grounding with Explicit Position-to-Coordinate Mapping, arXiv:2510.03230  
  https://arxiv.org/abs/2510.03230

这直接解释了为什么我们的 direct bbox 测试会全 0：直接要求通用 VLM 生成精确 pixel bbox 是最难形式。合理训练目标应先转为 candidate-based grounding。

## 3. 本项目到底解决什么问题

### 3.1 不是单纯视觉 CoT

如果只是让模型输出：

```text
我先观察图片，然后比较柱子高度，所以答案是 B。
```

这属于普通视觉 CoT。问题是它无法验证模型到底有没有看图，也无法判断“柱子高度”是否来自真实区域。

EviTool-VL 只接受可验证的视觉推理：

```json
{
  "thought": "需要比较 2021 年三个类别的柱高。",
  "action": {
    "tool": "detect",
    "args": {"mode": "bar", "max_results": 10}
  }
}
```

工具返回：

```json
{
  "ok": true,
  "evidence_id": "ev_001",
  "regions": [
    {"bbox": [120, 80, 160, 320], "label": "A"},
    {"bbox": [180, 50, 220, 320], "label": "B"}
  ]
}
```

最终输出：

```json
{
  "reasoning": [
    {
      "step": "ev_001 中 B 的柱高最高。",
      "evidence": ["ev_001"]
    }
  ],
  "answer": "B"
}
```

这里的关键不是“有 CoT”，而是 “CoT 中的每个关键断言都能追溯到真实 observation”。

### 3.2 不是泛泛 VLM Agent

泛泛 VLM Agent 更关注能不能调工具、能不能完成任务。EviTool-VL 关注的是：

- 小模型是否知道什么时候需要工具；
- 工具调用是否遵守 schema；
- 工具结果是否真的被后续推理使用；
- 最终答案是否引用了真实 evidence；
- 错误 OCR、误导 caption、相邻按钮会不会让模型走捷径；
- 工具成本是否可控。

因此更准确的任务名称是：

```text
Evidence-Grounded Visual Tool Reasoning
```

中文可写作：

```text
证据闭合的视觉工具推理
```

### 3.3 项目研究问题

RQ1：相比 direct answer，小模型使用本地视觉工具能否提升局部证据任务的准确率？

RQ2：相比普通 prompt-only tool use，SFT 后模型是否能稳定遵守 action-observation-final 协议？

RQ3：相比 answer-only GRPO，加入 evidence closure、grounding、OCR span 和 anti-text-bias reward 后，模型是否更少产生 unsupported reasoning？

RQ4：在 GUI grounding 中，candidate-based click/box selection 是否比 direct coordinate generation 更适合小 VLM？

RQ5：能否在 2 张 RTX 3090 和有限 API token 预算下跑通可复现的小模型 evidence-bound VLM agent 闭环？

## 4. 数据构建规模与 token 预算

### 4.1 当前数据源

已有 `eval_mini_100` 来自公开数据集：

| 数据集 | 当前用途 | 是否需要手写 |
|---|---|---|
| ChartQA | 图表读数、数值比较 | 否 |
| DocVQA | 文档 OCR、布局理解 | 否 |
| AI2D | 科学图示多选 | 否 |
| ScreenSpot-v2 | GUI grounding | 否 |

后续训练数据也应沿用公开数据源扩展，不手写基础 QA。我们需要构建的是工具轨迹和 evidence annotation。

### 4.2 推荐 v0 数据规模

不建议一开始构建 30K。原因：

- 100 万 token/model 的额度不足以做多轮、多模型、大规模 teacher generation；
- 工具轨迹质量比数量更关键；
- 当前最大短板是 GUI grounding，应该集中做 candidate/oracle traces；
- 没有经过 SFT smoke 前，30K 可能大量重复错误工具模式。

建议 v0：

| Split | 数量 | 说明 |
|---|---:|---|
| GUI grounding traces | 600-1000 | `detect(ui/text) -> candidate -> click/final` |
| ChartQA traces | 200-400 | `ocr/detect(bar)/measure -> final` |
| DocVQA traces | 200-400 | `ocr -> zoom/crop -> final` |
| AI2D traces | 100-200 | `inspect/crop/zoom -> final` |
| Hard negatives | 200-500 | 误导 caption、错误 OCR、相邻 GUI candidate |
| Total | 1300-2500 | v0 可训练规模 |

更保守的第一版可做 1K：

```text
GUI 500 + ChartQA 200 + DocVQA 200 + AI2D 100
```

### 4.3 API token 预算估算

根据此前百炼 direct smoke，每 8 条样本消耗约 6K-10K tokens，平均约：

```text
750-1300 tokens / sample
```

工具轨迹生成 prompt 更长，保守按：

```text
1500-2500 tokens / sample
```

估算每个 100 万 token 模型可支持：

```text
1000000 / 1500 ≈ 666 samples
1000000 / 2500 = 400 samples
```

如果只做短 prompt 初筛：

```text
1000000 / 1000 = 1000 samples
```

因此：

- 单模型 100 万 token：够做 400-1000 条 teacher-assisted traces；
- 7 个允许模型合计：理论上够做 2800-7000 条调用；
- 考虑重试、失败、修正、工具 observation 文本，稳妥 v0 是 1K-2K；
- 30K 需要千万级到数千万 token，不适合当前免费额度。

### 4.4 模型使用策略

严格只使用用户允许的模型，不额外尝试其他模型。

允许用于后续生成的模型：

- `qwen3.5-flash-2026-02-23`
- `MiniMax-M2.5`
- `qwen3.6-plus-2026-04-02`
- `glm-5.1`
- `qwen3.6-flash-2026-04-16`
- `qwen3.6-35b-a3b`
- `kimi-k2.6`

此前已做 smoke 的模型可作为历史参考：

- `qwen3.5-flash`
- `qwen3.5-27b`
- `qwen3.5-plus`
- `qwen3.5-122b-a10b`
- `qwen3.6-plus`

推荐策略：

1. 初筛和批量生成：优先用便宜快的 `qwen3.5-flash-2026-02-23` 或 `qwen3.6-flash-2026-04-16`。
2. 难例修正：使用 `qwen3.6-plus-2026-04-02`。
3. 多样性检查：少量使用 `glm-5.1`、`MiniMax-M2.5`、`kimi-k2.6`。
4. 不做全量多模型投票，只对 hard cases 做二次生成。
5. 所有 teacher 输出必须经过本地工具执行和规则过滤，不直接进入训练。

## 5. 训练数据格式

### 5.1 统一样本定义

输入样本：

```math
x = (I, q, m)
```

其中：

- `I`：图像；
- `q`：问题或 GUI instruction；
- `m`：任务元信息，如 task_type、choices、source。

工具轨迹：

```math
\tau = (a_1, o_1, a_2, o_2, ..., a_T, o_T, y)
```

其中：

- `a_t`：第 `t` 步模型输出 action；
- `o_t`：本地工具 observation；
- `y`：最终 JSON，包括 reasoning 和 answer。

每个 observation 生成 evidence：

```math
E(\tau) = \{e_1, e_2, ..., e_K\}
```

最终 reasoning 中引用的 evidence：

```math
C(y) = \{c_1, c_2, ..., c_M\}
```

证据闭合要求：

```math
C(y) \subseteq E(\tau)
```

### 5.2 Action JSON

```json
{
  "thought": "need candidate UI boxes",
  "action": {
    "tool": "detect",
    "args": {
      "mode": "ui",
      "max_results": 20
    }
  }
}
```

允许工具：

- `inspect`
- `crop`
- `zoom`
- `ocr`
- `detect`
- `measure`
- `click`
- `mark`，只用于调试，不进入训练答案流程

### 5.3 Observation JSON

```json
{
  "ok": true,
  "tool": "detect",
  "evidence_id": "ev_001",
  "regions": [
    {
      "candidate_id": "cand_001",
      "bbox": [920, 120, 980, 180],
      "center": [950, 150],
      "text": "close",
      "visual_type": "button"
    }
  ]
}
```

### 5.4 Final JSON

文本类：

```json
{
  "reasoning": [
    {
      "step": "ev_001 OCR 中显示总数为 14。",
      "evidence": ["ev_001"]
    }
  ],
  "answer": "14"
}
```

GUI 类：

```json
{
  "reasoning": [
    {
      "step": "ev_001 中 cand_003 与 instruction 描述的关闭按钮匹配。",
      "evidence": ["ev_001"]
    }
  ],
  "answer": [956, 158]
}
```

GUI 初期推荐输出 click point；如果需要 bbox，可输出：

```json
{
  "answer": [940, 140, 985, 200]
}
```

但训练优先级是：

```text
pointing > candidate selection > IoU@0.5 bbox
```

## 6. 数据构建流程

### 6.1 GUI oracle trace

对 ScreenSpot-v2，已知 GT bbox：

```math
b^* = [x_1,y_1,x_2,y_2]
```

先运行本地 `detect(ui/text)` 得到候选：

```math
B = \{b_1,b_2,...,b_N\}
```

候选命中定义：

```math
\text{hit}(b_i,b^*) =
\mathbb{1}[\text{IoU}(b_i,b^*) \ge 0.3]
\lor
\mathbb{1}[\text{center}(b_i)\in b^*]
```

选择最佳候选：

```math
i^* = \arg\max_i \left(\lambda_1 \text{IoU}(b_i,b^*) + \lambda_2 \mathbb{1}[\text{center}(b_i)\in b^*]\right)
```

如果 `max_i hit=1`，生成 oracle trace：

```text
detect(ui/text) -> click(candidate_id=i*) -> final
```

如果候选召回失败，则使用 GT bbox 生成 bootstrap trace：

```text
crop(gt_bbox expanded) -> click(center(gt_bbox)) -> final
```

这类样本标记为 `oracle_gt_injected=true`，后续可降低权重。

### 6.2 ChartQA trace

优先工具链：

```text
inspect -> ocr/detect(bar) -> measure/ocr -> final
```

过滤规则：

- final answer 必须与 GT 数值在 3% 相对误差内；
- 如果 answer 是类别名，OCR 或 chart region 中必须出现相关 label；
- 推理中的数值必须能在 observation 中找到，或由 measure 明确计算得到。

### 6.3 DocVQA trace

优先工具链：

```text
ocr(full/region) -> zoom/crop if needed -> final
```

过滤规则：

- final answer 必须与 GT exact/F1 匹配；
- answer 字符串或主要 token 必须出现在 OCR spans；
- 如果 OCR 未召回答案，不把 teacher 幻觉 trace 纳入正样本。

### 6.4 AI2D trace

优先工具链：

```text
inspect -> crop/zoom relevant region -> final option
```

过滤规则：

- final 可为选项字母或选项文本；
- reasoning 必须引用至少一个 visual evidence；
- 如果纯文本不看图也能答对，样本降权。

### 6.5 Hard negative

构造三类 hard negatives：

1. 误导 caption：

```text
附加文本说 A 最高，但图像真实最高为 B。
```

2. 错误 OCR：

```text
将关键 OCR 数字 62 替换成 52，要求模型用 crop/zoom 或图像证据验证。
```

3. 相邻 GUI candidate：

```text
提供多个相邻候选按钮，其中只有一个 center 落在 GT bbox。
```

这些 negative 不一定用于 SFT 正样本，但非常适合 RL reward 和鲁棒性评测。

## 7. 算法设计

### 7.1 总体训练阶段

```text
Stage 0: Direct / prompt-only baseline
Stage 1: Tool trace SFT
Stage 2: Evidence-closed RLVR/GRPO
Stage 3: Robustness and grounding evaluation
```

Stage 1 解决：

- 输出合法 JSON；
- 学会工具调用顺序；
- 学会引用 evidence id；
- GUI 学会从 candidate 中选择。

Stage 2 解决：

- 不只模仿 teacher；
- 在多个采样轨迹中偏好“答案正确且证据闭合”的轨迹；
- 惩罚 unsupported reasoning、错误 OCR 依赖、无效工具调用和过度工具调用。

### 7.2 Reward 定义

总奖励：

```math
R(\tau;x) =
w_a R_{\text{ans}}
+ w_g R_{\text{ground}}
+ w_e R_{\text{evidence}}
+ w_o R_{\text{ocr}}
+ w_t R_{\text{trace}}
+ w_r R_{\text{robust}}
- w_c C_{\text{tool}}
- w_h P_{\text{hallu}}
```

推荐初始权重：

| 项 | 权重 | 说明 |
|---|---:|---|
| `R_ans` | 0.35 | 最终答案正确性 |
| `R_ground` | 0.20 | bbox/point/candidate 命中 |
| `R_evidence` | 0.15 | evidence id 闭合和支持性 |
| `R_ocr` | 0.08 | OCR span 支持答案 |
| `R_trace` | 0.12 | JSON/schema/tool 合法性 |
| `R_robust` | 0.05 | 抗误导 caption/OCR |
| `C_tool` | 0.03 | 工具成本惩罚 |
| `P_hallu` | 0.02 | 幻觉实体/数字惩罚 |

权重不是论文创新点，必须通过 ablation 验证。

### 7.3 Answer reward

文本任务：

```math
R_{\text{ans}} =
\begin{cases}
1, & \text{EM}(\hat{y}, y^*)=1 \\
1, & \text{NumMatch}_{3\%}(\hat{y}, y^*)=1 \\
\text{F1}(\hat{y}, y^*), & \text{open short answer} \\
0, & \text{otherwise}
\end{cases}
```

数值匹配：

```math
\text{NumMatch}_{3\%} =
\mathbb{1}\left[
\frac{|\hat{v}-v^*|}{\max(|v^*|,1)} \le 0.03
\right]
```

AI2D 多选：

```math
R_{\text{ans}} =
\mathbb{1}[\hat{y}=y^*_{\text{letter}} \lor \hat{y}=y^*_{\text{text}}]
```

### 7.4 Grounding reward

GUI 输出 point：

```math
R_{\text{point}} =
\mathbb{1}[\hat{p} \in b^*]
```

GUI 输出 bbox：

```math
R_{\text{iou}} =
\min\left(\frac{\text{IoU}(\hat{b}, b^*)}{0.5}, 1\right)
```

候选选择：

```math
R_{\text{cand}} =
\mathbb{1}[\text{center}(b_{\hat{i}})\in b^*]
```

综合：

```math
R_{\text{ground}} =
\begin{cases}
R_{\text{point}}, & \text{if point output} \\
0.7R_{\text{iou}}+0.3\mathbb{1}[\text{center}(\hat{b})\in b^*], & \text{if bbox output} \\
R_{\text{cand}}, & \text{if candidate output}
\end{cases}
```

训练初期更鼓励 point/candidate，避免模型被严格 IoU 稀疏奖励卡死。

### 7.5 Evidence closure reward

证据闭合：

```math
R_{\text{closed}} =
\mathbb{1}[C(y)\subseteq E(\tau)]
```

证据非空：

```math
R_{\text{nonempty}} =
\mathbb{1}[|C(y)|>0]
```

证据支持性：

```math
R_{\text{support}} =
\frac{1}{M}\sum_{j=1}^{M}
\mathbb{1}[\text{claim}_j \text{ is supported by } e_{c_j}]
```

实际实现中，`supported` 先用规则近似：

- 数字是否出现在 OCR/measure observation；
- answer string 是否出现在 OCR spans；
- GUI candidate center 是否落在 GT bbox；
- 引用 bbox 是否包含目标区域；
- chart label 是否出现在 evidence region。

综合：

```math
R_{\text{evidence}} =
0.4R_{\text{closed}} + 0.2R_{\text{nonempty}} + 0.4R_{\text{support}}
```

### 7.6 Trace reward

```math
R_{\text{trace}} =
0.25R_{\text{json}}
+0.25R_{\text{schema}}
+0.25R_{\text{tool-valid}}
+0.25R_{\text{final-valid}}
```

其中：

- `R_json`：输出是否为合法 JSON；
- `R_schema`：字段是否符合 action/final 协议；
- `R_tool-valid`：工具名是否在 allowlist；
- `R_final-valid`：最终 answer 非空，GUI answer 可解析为 point/bbox/candidate。

### 7.7 OCR reward

对 DocVQA / ChartQA：

```math
R_{\text{ocr}} =
\mathbb{1}[\text{norm}(\hat{y}) \subseteq \text{OCRSpans}(E)]
```

如果答案是数值：

```math
R_{\text{ocr-num}} =
\mathbb{1}[\exists v \in \text{OCRNums}(E),
\frac{|v-\hat{v}|}{\max(|\hat{v}|,1)} \le 0.03]
```

综合：

```math
R_{\text{ocr}} = \max(R_{\text{ocr}}, R_{\text{ocr-num}})
```

### 7.8 Robust reward

对同一图像构造 clean prompt 和 biased prompt：

```math
x^{clean}=(I,q),\quad x^{bias}=(I,q,h)
```

其中 `h` 是误导 caption 或错误 OCR。

鲁棒性奖励：

```math
R_{\text{robust}} =
\mathbb{1}[\hat{y}^{bias}=y^*]
\cdot
\mathbb{1}[\text{evidence supports } \hat{y}^{bias}]
```

如果模型复述误导文本并忽略视觉证据：

```math
P_{\text{text-bias}} = 1
```

### 7.9 Tool cost

```math
C_{\text{tool}} =
\min\left(\frac{T}{T_{\max}},1\right)
```

默认：

```math
T_{\max}=3
```

如果 `T > 3`，额外惩罚：

```math
C_{\text{overuse}} = \max(0,T-3)
```

最终：

```math
C_{\text{tool-total}} = C_{\text{tool}} + 0.1 C_{\text{overuse}}
```

### 7.10 Hallucination penalty

```math
P_{\text{hallu}} =
\mathbb{1}[\text{reasoning cites nonexistent evidence}]
\lor
\mathbb{1}[\text{claim contains number not in evidence}]
\lor
\mathbb{1}[\text{claim names object/text not in evidence}]
```

更细粒度：

```math
P_{\text{hallu}} =
\frac{
N_{\text{unsupported-claims}}
}{
\max(N_{\text{claims}},1)
}
```

### 7.11 Evidence-Closed Tool GRPO objective

对每个输入 `x`，模型采样 `G` 条完整工具轨迹：

```math
\tau_i \sim \pi_\theta(\cdot|x),\quad i=1,\ldots,G
```

每条轨迹通过工具环境执行并打分：

```math
R_i = R(\tau_i;x)
```

组内 advantage：

```math
A_i =
\frac{R_i-\frac{1}{G}\sum_{j=1}^{G}R_j}
{\sqrt{\frac{1}{G}\sum_{j=1}^{G}(R_j-\mu_R)^2}+\epsilon}
```

token-level ratio：

```math
r_{i,t}(\theta)=
\frac{
\pi_\theta(y_{i,t}|x,y_{i,<t})
}{
\pi_{\theta_{\text{old}}}(y_{i,t}|x,y_{i,<t})
}
```

clipped objective：

```math
\mathcal{L}_{\text{GRPO}}(\theta)=
-\frac{1}{G}\sum_{i=1}^{G}
\frac{1}{|y_i|}\sum_{t=1}^{|y_i|}
\min\left(
r_{i,t}(\theta)A_i,
\text{clip}(r_{i,t}(\theta),1-\epsilon_c,1+\epsilon_c)A_i
\right)
+ \beta D_{\text{KL}}(\pi_\theta || \pi_{\text{ref}})
```

本项目的关键不在该 objective，而在：

```math
R(\tau;x)
```

也就是把本地工具执行、证据闭合、GUI grounding、OCR 支持和反文本偏置写成可验证 reward。

## 8. 技术栈

### 8.1 模型

Student：

- `Qwen/Qwen2.5-VL-3B-Instruct`：快速 SFT / smoke；
- `Qwen/Qwen3-VL-4B-Instruct`：主模型。

Teacher：

- 只使用用户明确允许的百炼/第三方模型池；
- teacher 不直接决定训练标签，必须经过工具执行和规则过滤。

### 8.2 训练框架

- PyTorch
- Transformers
- PEFT / LoRA / QLoRA
- bitsandbytes
- TRL `GRPOTrainer` 或自定义 GRPO loop
- LLaMA-Factory 可用于 SFT 快速启动

### 8.3 工具系统

已有本地工具：

- `inspect`
- `crop`
- `zoom`
- `ocr`
- `detect`
- `measure`
- `click`
- `mark`
- `trace`

需要增强：

- `detect(ui)` 返回稳定 candidate_id；
- 合并 OCR text boxes、UI contour boxes、icon-like boxes；
- 增加 candidate recall@k；
- observation 中显式包含 bbox、center、text、visual_type；
- GUI final 支持 candidate_id / point / bbox 三种输出。

### 8.4 数据处理

- 数据源：Hugging Face datasets / 本地缓存；

- JSONL 格式保存训练 trace；

文本任务：

## 10. 为什么这仍然有研究价值

已有工作确实做过 grounded CoT、step-wise reward、多模态 GRPO、GUI grounding RL。但 EviTool-VL 的价值在于把这些方向收敛到一个低算力、可执行、可审计的小模型训练闭环：

1. **从解释到执行**  
   不是 post-hoc 给 CoT 找区域，而是模型先 action，工具返回 observation，再 final。

2. **从最终答案到证据闭合**  
   不只看 answer accuracy，还看 evidence id 是否真实存在、是否支持推理步骤。

3. **从裸坐标到候选选择**  
   GUI grounding 不再强迫小 VLM 直接回归 pixel bbox，而是通过 detect candidates、crop/zoom、click point 降低任务难度。

4. **从大模型展示到小模型训练**  
   强模型只作为 teacher/plan proposer，小模型通过 SFT + RLVR 学会可复现行为。

5. **从单一 reward 到视觉工具环境 reward**  
   reward 不只是最终答案，而是把本地工具执行结果纳入训练信号。

推荐项目标题：

```text
EviTool-VL: Evidence-Closed Visual Tool Reasoning for Small Vision-Language Models
```

推荐一句话摘要：

```text
We train small VLMs to reason with executable visual tools by enforcing evidence closure between tool observations, intermediate rationales, and final answers.
```

中文摘要：

```text
本项目训练小参数 VLM 通过可执行视觉工具获取局部证据，并用证据闭合奖励约束工具观察、中间推理与最终答案的一致性，从而提升图表、文档和 GUI grounding 任务中的可靠性与可解释性。
```

## 11. 风险与应对

### 11.1 风险：teacher trace 幻觉

应对：

- teacher 只生成 action proposal；
- observation 必须由本地工具真实执行；
- final 必须过答案和 evidence filter；
- 不通过的 trace 不进正样本。

### 11.2 风险：OCR 质量差

应对：

- 多 OCR backend；
- 对 OCR 失败样本降权；
- 文档任务保留 OCR answer-in-observation 过滤；
- 不让模型学会引用不存在文字。

### 11.3 风险：GUI candidate recall 低

应对：

- 先统计 candidate recall@k；
- 合并 UI contour、OCR box、icon heuristic；
- recall 失败时用 GT expanded crop 做 bootstrap；
- 后续改进 detect(ui)。

### 11.4 风险：GRPO reward 稀疏

应对：

- 先 SFT；
- GUI 用 point-in-box 和 candidate reward；
- answer/evidence/trace 多项 dense reward；
- hard samples 分阶段加入；
- group size 从 4 开始，不盲目增大。

### 11.5 风险：模型学会无脑调用工具

应对：

- tool cost penalty；
- max tool steps；
- 对无需工具样本允许 direct final；
- 评测 avg_tool_calls 和 tool usefulness。

## 13. 最终建议

当前最务实的下一步不是继续跑更多 direct baseline，也不是直接做 30K 数据，而是：

1. 构建 500 条 GUI oracle/candidate traces；
2. 加上 500 条 ChartQA/DocVQA/AI2D evidence traces；
3. 先训 Qwen2.5-VL-3B LoRA 验证协议稳定；
4. 再训 Qwen3-VL-4B；
5. 最后做 200-500 条 hard negative 的 Evidence-Closed GRPO smoke。

如果 v0 能证明：

- protocol error 大幅下降；
- evidence closed rate 稳定；
- GUI pointing 明显提升；
- 文本类准确率不崩；

那么项目就有足够清晰的算法与工程价值，再扩到 5K-10K 数据。30K 应作为后续扩展目标，而不是当前免费额度下的第一阶段目标。
