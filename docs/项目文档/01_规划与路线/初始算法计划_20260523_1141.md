# EviTool-VL 算法计划

## 1. 项目定位

**项目名称**：EviTool-VL: Evidence-Grounded Visual Tool Reasoning for Small VLMs

**建议主攻方向**：小参数 VLM 的“证据约束视觉 CoT + 工具使用 + GRPO 后训练”。

这个方向比纯 Agentic Search、纯 RAG 或单纯复现 nanoGPT 更适合当前条件：

- 你有两张 RTX 3090，共 48GB 显存，适合做 3B-4B 级 VLM 的 QLoRA、SFT、DPO/GRPO，而不适合从零训练大模型。
- 你已经有 LangGraph Agent、RAG、Milvus/Neo4j、DPO、量化、ComfyUI、多模态数据构建经验，可以自然迁移到“VLM + 工具 + 后训练”。
- 2026 年 VLM 仍存在一个很具体的问题：模型会写很长的视觉 CoT，但推理步骤并不一定真的绑定图像证据，容易受文本先验、错误 OCR、误导 caption 影响。
- 该项目能同时体现算法能力、数据工程能力、训练能力和 Agent 工具调用能力，适合作为算法岗/多模态大模型岗的核心项目。

一句话项目描述：

> 基于 Qwen3-VL-4B 构建一个小算力可训练的视觉推理 Agent，使模型在回答图表、文档、GUI、空间推理问题时，先主动调用 crop、zoom、OCR、detect、measure 等视觉工具获取局部证据，再生成绑定证据的视觉 CoT，并通过 GRPO 奖励优化答案正确性、证据一致性和抗文本误导能力。

## 2. 背景问题

### 2.1 当前 VLM 的关键缺陷

2025-2026 年的 Qwen3-VL、Gemini、GPT-4o 类模型已经在文档理解、图表问答、GUI 操作和视觉数学上有明显进步，但小模型仍有几个稳定短板：

1. **视觉 grounding 不稳定**
   - 模型能够说出看似合理的推理链，但推理链中的关键实体、数字、位置关系未必来自图像真实区域。
   - 在文档、图表、GUI 截图任务中，模型经常根据问题文本猜答案，而不是回看局部视觉证据。

2. **长 CoT 漂移**
   - Thinking 模型倾向输出较长推理，token 成本高。
   - 视觉任务中，长推理并不等价于更强视觉验证，甚至会把早期错误不断放大。

3. **文本先验幻觉**
   - 当图片里有错误 OCR、误导 caption、相似按钮、相邻目标框时，模型容易相信文本提示或常识模式。
   - 例如图表问“2021 年哪个类别最高”，模型可能按常见趋势猜测，而不是定位柱状图区域。

4. **工具调用不可控**
   - VLM Agent 可以调用工具，但什么时候需要工具、调用什么工具、工具结果如何进入推理链，通常缺乏可训练的监督和奖励。
   - 现有 Visual Agent 更偏能力展示，算法层面的 reward 设计和 evidence 绑定仍有空间。

### 2.2 本项目要解决的问题

本项目不追求“让模型更大”，而是解决：

> 如何让 3B-4B 小 VLM 在有限显存下学会主动获取视觉证据，并让每一步推理都能被图像区域、OCR span 或工具返回结果验证？

对应研究问题：

- RQ1：视觉工具调用是否能提升小 VLM 在局部证据任务上的准确率？
- RQ2：证据绑定 CoT 是否能降低幻觉，提高 reasoning-answer alignment？
- RQ3：引入 hard negative 和 grounding reward 后，模型是否更抗错误 OCR、误导 caption 和文本先验？
- RQ4：能否在不显著增加推理 token 的情况下，获得更高的答案正确率和可解释性？

## 3. 技术路线总览

### 3.1 主模型与对照模型

主模型：

- `Qwen/Qwen3-VL-4B-Instruct`
- 用途：最终主实验模型。
- 原因：4B 规模适合双 3090 QLoRA；Qwen3-VL 系列强调 Visual Agent、空间感知、长上下文和多模态推理，方向匹配。

低算力对照：

- `Qwen/Qwen2.5-VL-3B-Instruct`
- 用途：baseline、快速调试、低显存 ablation。
- 原因：3B 模型更容易快速迭代，适合先验证数据格式、reward 函数和工具调用协议。

可选扩展：

- `Qwen/Qwen3-VL-2B-Instruct`：如果 4B GRPO 显存压力过大，可先用 2B 跑通完整闭环。
- `Qwen3-VL-4B-Thinking`：只做推理对照，不建议作为主训练对象，因为它的长 CoT 成本较高，且你的目标是训练一个更经济的 evidence-bound 推理模型。

### 3.2 框架选择

训练与推理框架：

- `PyTorch`
- `Transformers`
- `PEFT`
- `bitsandbytes`
- `TRL`
- `LLaMA-Factory`
- `vLLM` 或 `SGLang`

推荐分工：

- SFT 阶段：优先用 LLaMA-Factory 或 Transformers + PEFT。
- GRPO 阶段：用 TRL `GRPOTrainer`，自定义 reward functions。
- 批量推理与数据生成：用 vLLM 或 SGLang 部署 teacher/baseline。
- 工具系统：用 Python 实现本地视觉工具，后续可封装成 LangGraph 节点。

### 3.3 总体 pipeline

```text
原始图像任务数据
    ↓
OCR / detect / layout / chart parsing 预处理
    ↓
teacher 生成候选 visual tool trace
    ↓
规则校验：答案、证据框、OCR span、tool trace 是否闭合
    ↓
Difficulty-Aware Filtering
    ↓
EviTool-30K 训练集
    ↓
SFT: 学会工具调用格式和 evidence-bound CoT
    ↓
GRPO: 用答案正确性 + grounding 一致性 + 工具成本奖励优化
    ↓
评测：Answer / Grounding / Robustness / Efficiency / Forgetting
```

## 4. 算法创新点

### 4.1 Evidence-Bound Visual CoT

普通视觉 CoT：

```text
图中柱状图显示 A 比 B 高，所以答案是 A。
```

问题：这句话没有说明模型到底看了哪里，也无法验证“A 比 B 高”是否来自真实图像。

本项目要求模型输出证据绑定推理：

```json
{
  "thought": "需要比较 2021 年 A、B、C 三个类别的柱高。",
  "action": {
    "tool": "crop",
    "bbox": [120, 80, 520, 360]
  }
}
```

工具返回：

```json
{
  "evidence_id": "ev_1",
  "type": "crop",
  "bbox": [120, 80, 520, 360],
  "summary": "2021 年 A=42, B=57, C=35"
}
```

最终推理：

```json
{
  "reasoning": [
    {
      "step": "ev_1 显示 2021 年 B 的数值为 57，高于 A 的 42 和 C 的 35。",
      "evidence": ["ev_1"]
    }
  ],
  "answer": "B"
}
```

核心要求：

- 每个关键推理步骤必须引用 evidence id。
- evidence id 必须来自真实工具调用或数据集中标注的区域/OCR span。
- 没有证据支撑的推理步骤在训练奖励中扣分。

### 4.2 Grounding Consistency Reward

GRPO 阶段不用单一答案奖励，而是使用组合奖励：

```text
R = w_answer * R_answer
  + w_ground * R_grounding
  + w_ocr * R_ocr
  + w_trace * R_trace
  - w_tool * C_tool
  - w_hallu * P_hallucination
```

建议初始权重：

| Reward 项 | 权重 | 说明 |
|---|---:|---|
| `R_answer` | 0.40 | 最终答案 EM/F1/Accuracy |
| `R_grounding` | 0.25 | 证据框 IoU、pointing accuracy |
| `R_ocr` | 0.10 | 引用 OCR 文本是否真实存在 |
| `R_trace` | 0.15 | tool-call JSON 是否可解析、evidence 是否闭合 |
| `C_tool` | 0.05 | 工具调用过多扣分 |
| `P_hallucination` | 0.05 | 引用不存在实体、数字、区域扣分 |

奖励细节：

- 答案完全正确：`R_answer = 1`
- 数值题允许相对误差：`abs(pred - gt) / max(abs(gt), 1) <= 0.03`
- 多选题用 exact match。
- 开放式短答案用 normalized F1。
- grounding IoU 大于 0.5 给满分；0.3-0.5 给部分分。
- 引用不存在的 evidence id，直接扣 `R_trace`。
- 工具调用超过 3 次开始扣分，避免模型无脑调用工具。

### 4.3 Text-Bias Contrastive RL

构造 hard negatives，让模型学会“不轻信文字先验”：

1. **误导 caption**
   - 给图像附加错误描述，例如“图中最高柱是 A”，但真实最高柱是 B。
   - 奖励模型忽略错误 caption，通过视觉证据回答。

2. **错误 OCR**
   - 将 OCR 结果中的关键数字替换为相近数字。
   - 奖励模型通过 crop/zoom 或图像区域二次验证。

3. **相邻目标框**
   - 对 GUI、RefCOCO、ScreenSpot 样本，加入相邻按钮或相似目标。
   - 奖励模型定位正确区域，而不是选最近的文本。

4. **文本-only 可猜样本过滤**
   - 如果纯文本 LLM 不看图也能高置信答对，则降低样本权重。
   - 保留真正需要视觉证据的样本。

### 4.4 Difficulty-Aware Filtering

不是盲目堆 30K 数据，而是筛“有训练价值”的样本。

每条样本计算：

```text
difficulty_score =
    0.35 * baseline_error
  + 0.25 * tool_gain
  + 0.20 * grounding_need
  + 0.10 * trace_quality
  + 0.10 * hard_negative_sensitivity
```

保留规则：

- baseline 答错但 teacher + tool 能答对：优先保留。
- 无需视觉证据也能答对：降低权重或删除。
- teacher trace 不能闭合：删除。
- evidence 与答案冲突：删除。
- grounding 区域过大或过模糊：删除。

目标是从 10-15 万候选样本中筛出 2-5 万条高价值数据，最终形成 `EviTool-30K`。

## 5. 视觉工具设计

### 5.1 工具列表

| 工具 | 输入 | 输出 | 用途 |
|---|---|---|---|
| `crop` | bbox | cropped image + evidence id | 查看局部区域 |
| `zoom` | bbox, scale | zoomed crop + evidence id | 阅读小字、细节 |
| `ocr` | bbox optional | text spans + boxes | 文档、表格、图表文字 |
| `detect` | query optional | object boxes | 目标定位、GUI 元素检测 |
| `measure` | bbox/list of points | length/height/relative position | 图表、几何、空间关系 |
| `mark` | bbox/point | overlay image | 可视化证据，主要用于 demo |

### 5.2 Tool-call schema

模型输出动作统一为 JSON：

```json
{
  "thought": "我需要定位题目询问的图表区域。",
  "action": {
    "tool": "crop",
    "args": {
      "bbox": [120, 80, 520, 360]
    }
  }
}
```

工具返回统一为：

```json
{
  "evidence_id": "ev_001",
  "tool": "crop",
  "bbox": [120, 80, 520, 360],
  "content": {
    "ocr": [
      {
        "text": "B: 57",
        "bbox": [260, 210, 310, 240]
      }
    ],
    "summary": "局部区域显示 B 类别柱高最大"
  }
}
```

最终答案格式：

```json
{
  "reasoning": [
    {
      "step": "根据 ev_001，B 的数值为 57，高于 A 和 C。",
      "evidence": ["ev_001"]
    }
  ],
  "answer": "B"
}
```

## 6. 数据集构建

### 6.1 EviTool-30K 组成

| 类别 | 数据来源 | 数量目标 | 训练价值 |
|---|---|---:|---|
| 图表/表格/文档 | ChartQA、DocVQA、InfographicsVQA、AI2D | 10K | OCR、局部证据、数值比较 |
| STEM/空间推理 | MathVista、Geometry3K、visual puzzle | 8K | 几何、空间关系、视觉数学 |
| Grounding/计数/定位 | RefCOCO、VG-CoT 风格自动标注 | 8K | evidence region、目标框绑定 |
| GUI Agent | ScreenSpot、AndroidControl 截图子集 | 4K | 屏幕元素定位、工具调用 |

### 6.2 数据字段

每条样本保存为 JSONL：

```json
{
  "id": "evitool_chartqa_000001",
  "image": "images/chartqa/000001.png",
  "question": "Which category has the highest value in 2021?",
  "answer": "B",
  "task_type": "chart_qa",
  "evidence_regions": [
    {
      "id": "gt_ev_1",
      "bbox": [120, 80, 520, 360],
      "label": "2021 bar group"
    }
  ],
  "ocr_spans": [
    {
      "text": "B 57",
      "bbox": [260, 210, 310, 240]
    }
  ],
  "tool_trace": [
    {
      "thought": "The question asks for the highest category in 2021, so I should inspect the 2021 bar group.",
      "action": {
        "tool": "crop",
        "args": {
          "bbox": [120, 80, 520, 360]
        }
      },
      "observation": {
        "evidence_id": "ev_1",
        "summary": "B has value 57, higher than A and C."
      }
    }
  ],
  "hard_negatives": [
    {
      "type": "misleading_caption",
      "text": "The highest category in 2021 is A."
    }
  ],
  "difficulty": {
    "baseline_correct": false,
    "tool_gain": 1.0,
    "score": 0.82
  }
}
```

### 6.3 数据生成流程

1. 下载并标准化原始数据。
2. 对图像运行 OCR、layout detection、object detection。
3. 用 teacher 模型生成候选工具轨迹和 grounded CoT。
4. 规则校验：
   - answer 是否匹配 ground truth。
   - evidence bbox 是否覆盖关键 OCR 或目标。
   - 推理步骤是否引用存在的 evidence id。
   - 工具输出是否能支持最终答案。
5. 构造 hard negatives。
6. 运行 baseline，计算 difficulty score。
7. 过滤并导出 `train.jsonl`、`val.jsonl`、`test.jsonl`。

### 6.4 数据规模建议

第一阶段不要直接做 30K，建议分三档：

| 版本 | 数量 | 用途 |
|---|---:|---|
| `EviTool-1K` | 1K | 跑通格式、工具、SFT |
| `EviTool-5K` | 5K | 做第一版可报告实验 |
| `EviTool-30K` | 30K | 最终完整训练 |

## 7. 训练方案

### 7.1 Baseline 复现

先不训练，跑以下 baseline：

| Baseline | 说明 |
|---|---|
| Qwen2.5-VL-3B direct answer | 直接回答，不 CoT |
| Qwen2.5-VL-3B text CoT | 普通文字 CoT |
| Qwen3-VL-4B direct answer | 主模型直接回答 |
| Qwen3-VL-4B text CoT | 主模型普通 CoT |
| Qwen3-VL-4B prompt tool-use | 只靠 prompt 调工具，不训练 |

必须先得到 baseline 表，否则后续提升没有说服力。

### 7.2 SFT 阶段

目标：

- 让模型学会固定 JSON 输出。
- 让模型学会何时调用工具。
- 让模型学会引用 evidence id。
- 让模型学会输出简洁、可验证的视觉 CoT。

推荐参数：

| 参数 | 建议值 |
|---|---|
| model | `Qwen/Qwen3-VL-4B-Instruct` |
| precision | bf16/fp16 |
| quantization | 4-bit QLoRA |
| LoRA rank | 16 or 32 |
| LoRA alpha | 32 or 64 |
| LoRA dropout | 0.05 |
| learning rate | 1e-4 to 2e-4 |
| batch size | per_device 1 |
| gradient accumulation | 16-32 |
| epoch | 1-2 |
| max length | 4096 or 8192 |
| trainable modules | LLM attention/MLP + projector |
| frozen modules | vision tower |

显存策略：

- 开启 gradient checkpointing。
- 冻结 vision tower。
- image resolution 先控制在 448-768。
- 如果 4B OOM，先用 3B 跑通，再迁移到 4B。

### 7.3 GRPO 阶段

目标：

- 不只模仿 teacher trace，而是优化“答案正确 + 证据真实 + 工具经济”。
- 让模型在 hard negatives 下更愿意回看图像证据。

推荐设置：

| 参数 | 建议值 |
|---|---|
| trainer | TRL `GRPOTrainer` |
| samples per prompt | 4 |
| max completion length | 1024-2048 |
| reward funcs | answer, grounding, OCR, trace, tool cost |
| rollout model | 当前 LoRA 模型 |
| reference model | SFT checkpoint |
| train set | 5K-12K high-difficulty samples |
| eval interval | 每 200-500 steps |

如果 GRPO 不稳定，降级路线：

1. SFT + DPO：正样本为 evidence-bound trace，负样本为 hallucinated trace。
2. SFT + ORPO：更省显存，适合双 3090。
3. 只在 Qwen2.5-VL-3B 上做 GRPO，4B 只做 SFT+DPO。

### 7.4 训练产物

最终应产出：

- `EviTool-Qwen3VL-4B-LoRA`
- `EviTool-Qwen25VL-3B-LoRA`
- `EviTool-30K` 数据构建脚本和样本。
- reward function 代码。
- baseline 与 ablation 评测表。
- demo 页面或 Gradio 应用。

## 8. 评测方案

### 8.1 Answer 指标

| 任务 | 指标 |
|---|---|
| ChartQA | Accuracy / relaxed accuracy |
| DocVQA | ANLS / F1 |
| AI2D | Accuracy |
| MathVista-mini | Accuracy |
| ScreenSpot | Element accuracy |
| RefCOCO-style | pointing accuracy |

### 8.2 Grounding 指标

| 指标 | 说明 |
|---|---|
| Evidence IoU | 模型引用区域与 GT 区域 IoU |
| Evidence Recall | 答案所需关键证据是否被引用 |
| Pointing Accuracy | 预测点是否落在目标区域 |
| OCR Faithfulness | 引用文本是否真实出现在 OCR span |
| Reasoning-Answer Alignment | 推理链是否支持最终答案 |

### 8.3 Robustness 指标

构造三类干扰：

- misleading caption
- corrupted OCR
- adjacent distractor region

评测：

```text
robust_drop = clean_accuracy - attacked_accuracy
```

目标：

- baseline 在干扰下明显掉点。
- EviTool-VL 的 `robust_drop` 比 baseline 低 30%-50%。

### 8.4 Efficiency 指标

| 指标 | 目标 |
|---|---|
| average tool calls | <= 2.5 |
| average reasoning tokens | 低于 Qwen3-VL-4B-Thinking 的 70%-85% |
| invalid JSON rate | < 3% |
| evidence missing rate | < 5% |

### 8.5 Forgetting 指标

在通用 VLM 子集上评估：

- MMMU-mini
- MMBench subset
- TextVQA subset

目标：

- 通用能力相对下降不超过 2%。

## 9. Ablation 设计

必须做的消融：

| 实验 | 目的 |
|---|---|
| direct answer | 验证普通回答基线 |
| text CoT | 验证普通 CoT 是否足够 |
| prompt-only tool use | 验证不训练只 prompt 的上限 |
| SFT only | 验证模仿学习收益 |
| SFT + GRPO(answer only) | 验证只有答案奖励是否会产生伪 CoT |
| SFT + GRPO(answer + grounding) | 验证 grounding reward 收益 |
| SFT + GRPO(full reward) | 验证完整算法 |
| full reward without hard negatives | 验证 Text-Bias Contrastive RL 的必要性 |
| Qwen2.5-VL-3B vs Qwen3-VL-4B | 验证模型规模与新架构收益 |

预期结果：

- prompt-only tool use 会有提升，但 JSON 无效率和工具滥用更高。
- answer-only GRPO 可能提高答案，但 grounding 不一定提高。
- full reward 应在 robust setting 和 evidence alignment 上显著领先。

## 10. 12 周执行计划

| 周期 | 目标 | 具体任务 | 产出 |
|---|---|---|---|
| 第 1 周 | Baseline 环境 | 配置 Transformers、TRL、LLaMA-Factory；跑 Qwen2.5-VL-3B 和 Qwen3-VL-4B 推理；整理小评测集 | baseline 推理脚本、初始结果表 |
| 第 2 周 | 视觉工具 v1 | 实现 crop、zoom、OCR、mark；统一 tool-call schema；设计 evidence id 机制 | `tools/` 模块、工具 JSON 协议 |

## 12. 实验目标

### 12.1 最低可交付目标

完成以下内容即可作为一个有说服力的算法项目：

- `EviTool-5K` 数据集。
- Qwen2.5-VL-3B SFT + GRPO。
- Qwen3-VL-4B SFT。
- 至少 3 个 benchmark 的结果。
- 至少 4 组 ablation。
- 一个可视化 demo。

### 12.2 理想目标

完整版本应达到：

- `EviTool-30K` 数据集。
- Qwen3-VL-4B SFT + GRPO。
- 在 ChartQA、DocVQA、AI2D、MathVista-mini、ScreenSpot 上完成主实验。
- 相比 Qwen3-VL-4B text CoT，在局部证据强相关任务上提升 5-10 个点。
- 在误导 caption / 错误 OCR 场景下，性能下降幅度减少 30%-50%。
- 平均工具调用次数不超过 2.5。
- 输出 evidence trace，可视化证据框。

## 13. 风险与备选方案

| 风险 | 表现 | 备选方案 |
|---|---|---|
| 4B GRPO 显存不足 | OOM、速度过慢 | 先 3B GRPO，4B 做 SFT+DPO |
| JSON 输出不稳定 | 工具调用无法解析 | SFT 阶段增加格式数据，reward 加 parse penalty |
| teacher trace 质量差 | 推理链看似合理但证据不闭合 | 强规则过滤，保留少量高质量数据 |
| 工具调用过多 | 推理慢、token 多 | 加 tool cost reward，限制最多 3 次工具 |
| OCR 噪声影响训练 | 文档任务不稳定 | OCR span 置信度过滤，加入 corrupted OCR robustness |
| 提升只体现在小集合 | 泛化不足 | 增加多任务数据比例，做 held-out dataset |

## 14. 简历表述建议

项目名称：

> EviTool-VL：基于 Qwen3-VL-4B 的证据约束视觉工具推理后训练框架

简历 bullet：

- 面向小参数 VLM 在图表、文档、GUI 和空间推理任务中的弱 grounding 与视觉幻觉问题，设计 Evidence-Bound Visual CoT 机制，使模型在推理过程中显式调用 crop/OCR/detect/measure 等视觉工具，并将每一步推理绑定至可验证 evidence id。
- 构建 EviTool-30K 多任务训练集，融合 ChartQA、DocVQA、AI2D、MathVista、RefCOCO、ScreenSpot 等数据，设计 difficulty-aware filtering 与 hard negative 构造策略，过滤纯文本可猜样本并增强抗误导 caption / 错误 OCR 能力。
- 基于 Qwen3-VL-4B-Instruct 进行 QLoRA SFT 与 GRPO 后训练，设计由答案正确性、证据 IoU、OCR 一致性、trace 可解析性和工具调用成本组成的多目标 reward，提升小 VLM 的可解释视觉推理能力。
- 在 ChartQA、DocVQA、AI2D、MathVista-mini、ScreenSpot 等 benchmark 上进行 answer、grounding、robustness、efficiency 多维评测，并通过 ablation 验证 grounding reward、hard negatives 和 tool-use training 的有效性。
