# EviTool-VL 阶段总结与未来计划

生成时间：2026-05-25

## 1. 项目定位

EviTool-VL 的目标是研究“小型视觉语言模型能否通过显式视觉工具调用实现证据闭合的视觉推理”。当前重点不是单纯提升最终答案准确率，而是把模型行为拆成可审计的 action-observation-final 流程：模型先选择本地视觉工具，工具返回带 evidence id 的结构化 observation，模型再输出引用证据的最终答案。

项目覆盖四类任务：

- ChartQA：图表、表格、颜色和数值读数。
- DocVQA：文档和图像文字问答。
- AI2D：科学图示多选问答。
- ScreenSpot-v2：GUI 元素定位和点击。

当前主要模型是：

- `/root/models/Qwen2.5-VL-3B-Instruct`
- `/root/models/Qwen3-VL-4B-Instruct`

仓库位置：`/root/Workspace/VLM/EviTool-VL`。

## 2. 已完成的工程基础

### 2.1 本地工具层

已经实现并统一到 JSON action/observation 协议的工具包括：

- `inspect`：图像尺寸和基础统计。
- `crop` / `zoom`：局部裁剪和放大。
- `ocr`：EasyOCR、PaddleOCR、Tesseract 及 fallback 候选文本区域。
- `detect`：layout、text、ui、bar、color、diagram 检测。
- `measure`：bbox、中心点、IoU、距离、柱高等测量。
- `mark` / `visualize`：可视化框和点。
- `click`：虚拟 GUI 点击或目标框记录。
- `trace`：执行多步工具轨迹。

统一执行入口是 `tools/runner.py`。所有工具返回 `evidence_id`，为 evidence closed 评估和 RL reward 提供基础。

### 2.2 Prompt-only 工具调用评测

实现了 `eval/eval_tool_baseline.py`，支持：

- 模型输出 JSON action。
- 程序执行本地工具。
- observation 追加回上下文。
- 模型输出 final JSON。
- 统计 final parse、protocol error、tool success、evidence reference、evidence closed、avg tools 等指标。

早期 prompt-only 结果说明：

- 4B 的工具协议更稳定，evidence closed 可达到较高水平。
- 3B 容易直接回答、遗漏 evidence id、工具协议不稳定。
- prompt-only 能改善 GUI pointing，但还不能可靠输出紧框。

### 4.3 候选源 A/B 消融

报告路径：`/root/Workspace/VLM/EviTool-VL/reports/tool_candidate_source_ablation.md`

核心结论：

| Variant | Recall@30 | Oracle GT 注入估计 | Avg Best IoU | Avg Hit Rank |
|---|---:|---:|---:|---:|
| 初始 detect(ui)，无 OCR fallback | 62.20% | 37.80% | 0.2906 | 10.97 |
| detect(ui) + OCR fallback | 80.20% | 19.80% | 0.3561 | 9.09 |

OCR fallback 带来 +18.00% Recall@30，是目前最明确有效的增益来源。

候选源结论：

- `text_expanded`、`ocr_text` 是最关键来源。
- `query_prior` 对 close/minimize/settings 等 GUI 先验有帮助，但可能带来大量高分先验框，需要 reranker。
- `ui_rect`、`ui_edge`、`icon_visual`、`layout` 单独召回有限，但作为补充来源不应删除。

## 5. SFT v1 与 v2

### 5.1 SFT v1

第一版 SFT 使用 1k trace 转换得到的混合数据，包含 direct retention、工具轨迹和 GUI 目标。

主要观察：

- 3B direct GUI pointing 提升明显：从 21.50% 到 40.00%。
- 3B tool 模式反而学会大量绕过工具，avg tools 降到 0.42，protocol error 仍高。
- 4B tool 模式 evidence closed 很高，但 text/tool accuracy 有轻微负迁移。
- 说明简单混合 direct、tool、GUI bbox 回归不是理想训练目标。

## 7. RL 准备工作

当前已经完成 RL 前置环境，不包含真实 RL 训练。

新增文件：

- `rl/rollout_env.py`：工具交互 rollout 环境。
- `rl/rewards.py`：可解释 reward 分解。
- `scripts/rl_rollout_smoke.py`：不加载模型的 rollout dry-run。
- `scripts/run_trl_grpo_smoke.py`：TRL GRPO 小样本 smoke 脚本。

Reward 分解包括：

- answer reward：文本 relaxed/exact 或 GUI pointing/IoU。
- evidence reward：是否引用真实存在的 evidence id，是否 evidence closed。
- format reward：final JSON、协议和 parse 是否正确。
- tool reward：工具是否执行成功。
- cost penalty：工具调用过多扣分。
- empty penalty：空答案扣分。

已验证：

- `py_compile` 通过。
- rollout dry-run 能执行工具、闭合 evidence id、输出 reward。
- TRL GRPO dry-run 能验证数据和 reward plumbing。

真实 GRPO 尚未启动，原因是当前两张 GPU 正在跑 SFT v2 tool eval。下一步等 GPU 空出后运行：

## 8. 当前主要问题

### 8.1 SFT v2 tool 结果未完成

当前不能判断 SFT v2 是否改善 tool 模式。Direct 结果说明 v2 没有提升 direct GUI grounding，但 v2 本来主要目标是 tool/gui_select，不是 direct。

### 8.2 GUI 召回达到中期目标，但还未达到论文理想目标

当前 Recall@30 80.20%、Oracle GT 注入 19.80%。中期目标已达成，但距离 Recall@30 85%+ 和 Oracle 15% 以下还有差距。

### 8.3 GUI 定位需要从自由 bbox 回归转向 candidate selection/reranking

模型直接生成 bbox 的稳定性弱，尤其 4B direct 常能输出格式但坐标不准。更合理路径是：detect 生成 top-K 候选，模型输出 candidate_id 或 click point，程序再映射到 bbox。

### 8.4 AI2D 强证据样本不足

AI2D strong evidence rate 仍低，当前强证据样本主要来自选项文本或标签 OCR 能定位的样本。后续需要对象框、箭头边、标签关系等更细粒度证据结构。

### 8.5 ChartQA 不能只依赖 OCR

ChartQA answer evidence position 只有 46.00%。很多答案来自图表结构或数值估计，需要更强 chart parser、axis/legend/bar/line 结构识别和数值反推。

### 8.6 RL 还缺真正交互式训练闭环

当前只是 rollout/reward dry-run 和 TRL GRPO plumbing 准备。真正用于论文研究的 RL 需要支持多轮工具调用、环境反馈 masking、credit assignment、KL/reference model、reward ablation 和失败案例审计。

   - Text relaxed、GUI pointing、GUI IoU@0.5、evidence closed、avg tools、protocol error。
4. 判断 SFT v2 是否值得作为 RL init。

### 阶段 B：OmniParser A/B

1. 下载 OmniParser v2 权重到本地，不提交权重。
2. 跑完整候选 A/B。
3. 如果有效，将 OmniParser 作为可选 provider 融入 trace 构建。
4. 如果无效，不进入主训练 pipeline，只保留为可复现实验。

### 阶段 C：RL smoke

1. 用 Qwen2.5-VL-3B + SFT v2 LoRA 作为初始 policy。
2. 跑 8-32 条样本的 TRL GRPO smoke。
3. 检查：reward 是否稳定、loss 是否正常、输出是否退化、工具协议是否崩。
4. 不追求分数提升，只验证训练闭环安全。

### 阶段 D：交互式工具 RL

1. 把 `rollout_env.py` 接入 TRL `rollout_func` 或 `environment_factory`。
2. 让模型在训练中真实执行 action-observation-final。
3. 对 reward 做 ablation：answer-only、answer+format、answer+evidence、answer+evidence+tool-cost。
4. 加入 GUI candidate-level reward：candidate_id hit、pointing、IoU、rank-aware reward。
5. 对 hard/weak 样本做小规模 teacher proposal，但必须经过本地工具执行和 GT 过滤，不能直接相信 teacher chain-of-thought。

### 阶段 E：迁移到 verl 做算法研究

TRL 适合小样本 smoke 和快速验证。若要研究 GRPO/GIGPO/T2PO/credit assignment 等算法，应迁移到 verl，因为 verl 更适合大规模异步 rollout、多模态 RL、多轮 tool calling 和算法改造。

迁移前提：

- reward 已稳定。
- rollout 环境已验证。
- 小样本 GRPO 不崩。
- 有明确的算法变量和 ablation 设计。

### 阶段 F：扩展数据到 2k-3k

扩展原则：

- 不直接扩大弱样本。
- 优先扩充高质量 strong evidence 样本。
- GUI 只纳入 candidate recall 命中且非 oracle 的样本。
- AI2D 重点补对象/箭头/标签关系证据。
- ChartQA 重点补结构化 chart evidence。
- 对 hard/weak 样本可以调用强模型生成 trace proposal，但必须本地工具执行、答案正确、evidence closed、grounding 命中后才能入库。
