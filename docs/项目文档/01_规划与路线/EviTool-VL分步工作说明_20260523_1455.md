# EviTool-VL 分步工作说明

本文档基于 `算法计划.md`，目标是把 EviTool-VL 从想法拆成可以执行、可以验收、适合双 RTX 3090 的工程路线。

## 0. 你现在最开始应该做什么

你现在有两张 RTX 3090，不要一开始就做 Qwen3-VL-4B GRPO，也不要先构建 30K 数据。第一阶段目标只有一个：

> 用 3B/4B 模型跑通“环境 -> 推理 -> baseline -> 工具调用格式 -> 1K 数据 -> SFT 小闭环”。

建议前 3 天只做下面 6 件事：

1. 确认机器状态：CUDA、驱动、两张 3090、磁盘空间、Python 环境。
2. 拉代码：Qwen3-VL、LLaMA-Factory、TRL，并新建自己的 `EviTool-VL` 项目仓库。
3. 下载或缓存模型：先 Qwen2.5-VL-3B-Instruct，再 Qwen3-VL-4B-Instruct。
4. 跑通最小推理脚本：单图问答、JSON 输出、固定 seed。
5. 选 100 条小评测集：ChartQA/DocVQA/AI2D/ScreenSpot 各取一部分。
6. 记录 baseline 表：direct answer、text CoT、prompt-only tool-use。

只有 baseline 可复现之后，后面的算法改进才有说服力。

## 2. 环境配置

### 2.1 系统检查

先检查 GPU 和驱动：

```bash
nvidia-smi
nvcc --version
df -h
```

双 3090 的现实约束：

- 总显存是 48GB，但不是一张 48GB 卡；训练时要考虑 DDP/ZeRO/QLoRA 的切分方式。
- 4B QLoRA SFT 是合理目标。
- 4B GRPO 有风险，优先 3B GRPO 跑通。
- 视觉分辨率先控制在 448-768，max length 先用 4096。
- 数据、模型和中间 checkpoint 至少预留 500GB 磁盘，理想是 1TB 以上。

### 2.2 Conda 环境

推荐 Python 3.10 或 3.11。先用一个主环境跑通 baseline：

```bash
conda create -n evitool python=3.10 -y
conda activate evitool
pip install --upgrade pip
```

安装 PyTorch 时按你的 CUDA 版本选择。常见 CUDA 12.1：

```bash
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu121
```

安装基础包：

```bash
pip install "transformers>=4.57.0" accelerate peft bitsandbytes datasets evaluate
pip install qwen-vl-utils pillow opencv-python matplotlib pandas numpy scipy scikit-learn
pip install trl
pip install gradio jsonschema tqdm rich
```

如果后续使用 LLaMA-Factory，再按它的仓库说明单独安装。不要一开始把 vLLM、SGLang、FlashAttention、LLaMA-Factory 全混在一个不可控环境里；先把最小推理和 QLoRA SFT 跑通。

### 2.3 可选加速项

3090 是 Ampere 架构，可以考虑：

```bash
pip install flash-attn --no-build-isolation
```

如果编译失败，不要卡在这里。先不用 FlashAttention 跑通流程，后面再优化速度。

## 5. 第一阶段：Baseline

### 5.1 为什么 baseline 必须先做

你的项目核心不是“我训练了一个模型”，而是证明：

- 工具调用比直接回答强。
- evidence-bound CoT 比普通 CoT 更可靠。
- grounding reward 比 answer-only reward 更抗幻觉。
- hard negatives 能降低文本先验依赖。

这些都需要 baseline 表支撑。

### 5.2 先跑 5 个 baseline

最小 baseline：

| 编号 | 模型 | 设置 | 目的 |
|---|---|---|---|
| B1 | Qwen2.5-VL-3B | direct answer | 低算力基准 |
| B2 | Qwen2.5-VL-3B | text CoT | 普通推理基准 |
| B3 | Qwen3-VL-4B | direct answer | 主模型基准 |
| B4 | Qwen3-VL-4B | text CoT | 主模型普通 CoT |
| B5 | Qwen3-VL-4B | prompt-only tool-use | 不训练工具调用上限 |

第一版评测集不要大，先做 `eval_mini_100.jsonl`：

- ChartQA 30 条
- DocVQA 25 条
- AI2D 20 条
- ScreenSpot 15 条
- RefCOCO/定位类 10 条

每条样本保存：

```json
{
  "id": "chartqa_0001",
  "image": "data/raw/chartqa/0001.png",
  "question": "Which category has the highest value in 2021?",
  "answer": "B",
  "task_type": "chart_qa",
  "meta": {}
}
```

### 5.3 baseline 验收标准

第 1 周结束必须有：

- `scripts/run_infer.py`
- `eval/eval_answer.py`
- `reports/baseline_table.md`
- 每个 baseline 的原始输出 JSONL
- 20 个失败案例截图或样本记录

## 6. 第二阶段：视觉工具 v1

先做 4 个工具：

| 工具 | 第一版实现 |
|---|---|
| `crop` | PIL/OpenCV 裁剪 bbox |
| `zoom` | bbox 裁剪后 resize |
| `ocr` | PaddleOCR 或 EasyOCR，后续可替换 |
| `mark` | 在原图上画框，给 demo 和人工检查用 |

统一 action schema：

```json
{
  "thought": "需要查看图表中 2021 年的柱状区域。",
  "action": {
    "tool": "crop",
    "args": {
      "bbox": [120, 80, 520, 360]
    }
  }
}
```

统一 observation schema：

```json
{
  "evidence_id": "ev_001",
  "tool": "crop",
  "bbox": [120, 80, 520, 360],
  "content": {
    "summary": "该区域包含 2021 年 A/B/C 三组柱。",
    "ocr": []
  }
}
```

工具层不要依赖模型。工具输入 JSON，输出 JSON，便于后续训练、评测和 demo 复用。

## 7. 第三阶段：EviTool-1K 数据闭环

不要直接做 30K。先做 `EviTool-1K`，比例建议：

| 类别 | 数量 |
|---|---:|
| ChartQA/图表 | 350 |
| DocVQA/文档 | 250 |
| AI2D/STEM | 200 |
| ScreenSpot/GUI | 100 |
| RefCOCO/定位 | 100 |

每条样本需要同时满足：

- 有原图和问题。
- 有标准答案。
- 有 evidence region 或 OCR span。
- 有至少一条可解析 tool trace。
- 最终 reasoning 引用的 evidence id 真实存在。

训练样本格式：

```json
{
  "id": "evitool_chartqa_000001",
  "image": "images/chartqa/000001.png",
  "question": "Which category has the highest value in 2021?",
  "answer": "B",
  "task_type": "chart_qa",
  "evidence_regions": [
    {"id": "gt_ev_1", "bbox": [120, 80, 520, 360], "label": "2021 bar group"}
  ],
  "tool_trace": [
    {
      "thought": "The question asks for the highest category in 2021, so I should inspect the 2021 bar group.",
      "action": {"tool": "crop", "args": {"bbox": [120, 80, 520, 360]}},
      "observation": {"evidence_id": "ev_1", "summary": "B has value 57, higher than A and C."}
    }
  ],
  "final": {
    "reasoning": [
      {"step": "ev_1 shows B has the highest value.", "evidence": ["ev_1"]}
    ],
    "answer": "B"
  }
}
```

## 8. 算法改进落地顺序

### 8.1 先做 Evidence-Bound Visual CoT

第一版算法创新只强制三件事：

1. 模型必须用 JSON action 调工具。
2. 工具返回必须产生 `evidence_id`。
3. 最终 reasoning 每个关键步骤必须引用 evidence。

不要一开始就追求复杂 reward。SFT 阶段先让格式稳定。

### 8.2 再做 Difficulty-Aware Filtering

每条候选样本计算：

```text
difficulty_score =
    0.35 * baseline_error
  + 0.25 * tool_gain
  + 0.20 * grounding_need
  + 0.10 * trace_quality
  + 0.10 * hard_negative_sensitivity
```

实际第一版可以简化：

- baseline 答错且 tool trace 答对：保留。
- baseline 答对且不需要视觉证据：降权或删除。
- trace JSON 不可解析：删除。
- evidence id 不闭合：删除。
- bbox 过大，例如覆盖全图 80% 以上：删除或降权。

### 8.3 再做 Text-Bias Contrastive RL

hard negatives 先做三类：

- misleading caption：附加错误图片描述。
- corrupted OCR：替换关键数字或实体。
- adjacent distractor：GUI/定位任务加入相邻干扰框。

这部分不要急着进 RL，先用于评测集和 DPO 负样本构造。

### 8.4 最后做 GRPO

GRPO 是第 8-9 周任务，不是第一周任务。推荐先在 3B 上实现：

```text
R = 0.40 * R_answer
  + 0.25 * R_grounding
  + 0.10 * R_ocr
  + 0.15 * R_trace
  - 0.05 * C_tool
  - 0.05 * P_hallucination
```

第一版只启用：

- `R_answer`
- `R_trace`
- `C_tool`

第二版再加入：

- `R_grounding`
- `R_ocr`
- `P_hallucination`

这样更容易定位训练不稳定的来源。

## 9. 训练路线

### 9.1 SFT v1：Qwen2.5-VL-3B

目的：验证数据格式、collator、JSON 输出和 LoRA 训练。

建议参数：

| 参数 | 建议 |
|---|---|
| model | `Qwen/Qwen2.5-VL-3B-Instruct` |
| quantization | 4-bit QLoRA |
| LoRA rank | 16 |
| LoRA alpha | 32 |
| per_device_train_batch_size | 1 |
| gradient_accumulation_steps | 16 |
| max_length | 4096 |
| image resolution | 448-672 |
| epoch | 1 |
| vision tower | frozen |

验收：

- invalid JSON rate < 10%。
- evidence missing rate < 15%。
- eval_mini 上相对 direct answer 有提升。

### 9.2 SFT v2：Qwen3-VL-4B

目的：主模型 SFT。

建议参数：

| 参数 | 建议 |
|---|---|
| model | `Qwen/Qwen3-VL-4B-Instruct` |
| quantization | 4-bit QLoRA |
| LoRA rank | 16 或 32 |
| LoRA alpha | 32 或 64 |
| per_device_train_batch_size | 1 |
| gradient_accumulation_steps | 16-32 |
| max_length | 4096，稳定后 8192 |
| image resolution | 448-768 |
| epoch | 1-2 |
| vision tower | frozen |

双 3090 先用 DDP/accelerate。OOM 时按顺序降级：

1. 降 image resolution。
2. 降 max_length。
3. LoRA rank 从 32 降到 16。
4. gradient checkpointing 必须打开。
5. 先训练 projector + LLM attention，vision tower 继续冻结。

### 9.3 GRPO v1：先 3B

目的：验证 reward 是否正确，而不是追求最高分。

建议：

- samples per prompt = 4
- max completion length = 1024
- train set = 1K-5K high difficulty
- eval interval = 200 steps
- 先不用 vLLM rollout，稳定后再加速

如果 3B GRPO 都不稳定，不要迁移 4B。先修 reward、格式和样本质量。

## 10. 评测与报告

每次训练后都生成同一套表：

| 指标 | 说明 |
|---|---|
| answer accuracy | 最终答案正确率 |
| relaxed accuracy | 数值题允许 3% 相对误差 |
| invalid JSON rate | action/final JSON 解析失败比例 |
| evidence missing rate | reasoning 引用不存在 evidence 的比例 |
| evidence IoU | 预测 evidence 与 GT bbox 的 IoU |
| average tool calls | 平均工具调用次数 |
| average output tokens | 平均输出长度 |
| robust drop | clean accuracy - attacked accuracy |

必须保留失败案例：

- 答案对但证据错。
- 证据对但答案错。
- JSON 格式错。
- 工具调用过多。
- 被 misleading caption 带偏。
- OCR 数字读错。

这些失败案例会直接变成报告亮点，因为它们能说明你知道算法边界在哪里。

## 12. 你没提到但必须做的部分

### 12.1 数据版本管理

每个数据版本都要有：

- 生成脚本 commit id。
- 原始数据来源。
- 样本数量。
- 过滤规则。
- 随机种子。
- 训练/验证/测试切分文件。

否则后面结果无法复现。

### 12.2 日志和实验追踪

建议至少保存：

- config yaml
- git commit hash
- model name
- dataset version
- train loss
- eval metrics
- 失败样本 JSONL

可以用 WandB，也可以先用本地 CSV/Markdown。

### 12.3 Prompt 和输出协议冻结

SFT 前必须冻结一版协议：

- action JSON 格式。
- observation JSON 格式。
- final answer JSON 格式。
- 最多工具调用次数。
- bbox 坐标系统，是像素坐标还是归一化坐标。

建议第一版用像素坐标，后续再兼容归一化坐标。

### 12.4 License 和数据合规

每个公开数据集都检查 license。项目可以公开代码，但数据可能只能公开构建脚本和少量样例。

### 12.5 显存和速度基准

每个训练配置记录：

- peak GPU memory
- tokens/sec
- images/sec
- step time
- checkpoint size

这能帮助你决定 4B 是否值得做 GRPO。

## 14. 当前决策建议

以你的两张 3090，最合理的起步路线是：

1. 第 1 周：只做环境、模型推理、100 条 baseline。
2. 第 2-4 周：做工具和 `EviTool-1K`。
3. 第 6 周：先训 Qwen2.5-VL-3B SFT。
4. 第 7 周：再训 Qwen3-VL-4B SFT。
5. 第 8 周以后：只在 3B 上先做 GRPO。
6. 4B GRPO 作为理想目标，不作为最低可交付目标。

最低可交付项目应锁定：

- `EviTool-5K`
- Qwen2.5-VL-3B SFT + GRPO
- Qwen3-VL-4B SFT
- 3 个 benchmark
- 4 组 ablation
- 一个可视化 demo

这样最稳，也最符合双 3090 的算力边界。
