## 关键技术名词

- `VLM`：Vision-Language Model，视觉语言模型。这里指能同时理解图像和文本的模型，例如 Qwen2.5-VL、Qwen3-VL。
- `GUI`：Graphical User Interface，图形用户界面。网页、桌面软件、表单、菜单、表格都属于 GUI 场景。
- `Agentic RL`：智能体强化学习。模型不是只回答一个静态问题，而是在环境中连续观察、行动、获得奖励，并逐步学会完成任务。
- `BrowserRL`：浏览器强化学习环境。这里指我们用本地 HTML 任务和 Playwright 组成的可 reset、可 step、可 verifier 自动判分的浏览器 GUI 环境。
- `Playwright`：浏览器自动化工具。它能用程序打开网页、点击、输入、截图；我们用它把模型输出的动作真正执行到浏览器里。
- `reset/step/observation/reward`：强化学习环境接口。`reset` 是重置任务，`step` 是执行一步动作，`observation` 是截图等观察，`reward` 是动作得到的奖励。
- `Verifier`：验证器。它用程序检查页面状态，例如表单是否填对、目标项是否选中，并返回 success 和 reward。
- `Rollout`：一次完整任务尝试。从 reset 开始，模型多步执行动作，直到成功、失败或到达最大步数。
- `Tool-call trajectory`：工具调用轨迹。每一步都是一个可执行动作，例如 `click`、`type`、`press`、`scroll`，多步组成完整任务。
- `History-aware trajectory SFT`：带历史信息的轨迹监督微调。训练时给模型当前截图、目标、历史动作、verifier progress，让它输出下一步 action。
- `LoRA adapter`：低秩适配增量权重。base model 不直接改全量参数，只训练少量 LoRA 参数。这里的 adapter 是泛称，LoRA 是具体 adapter 类型。
- `Current model`：当前策略模型。现阶段指 `/root/models/Qwen2.5-VL-3B-Instruct` 加上当前 LoRA adapter。
- `On-policy RL`：在线策略强化学习。训练数据来自当前模型自己在环境中采样的动作，而不是只用 scripted oracle 专家轨迹。
- `GRPO`：Group Relative Policy Optimization。对同一状态采多个动作，用组内 reward 相对高低计算优势，让模型更偏向高 reward 动作。
- `Trainable group`：可训练动作组。同一状态下多次采样动作，如果 reward 有差异，就能提供 RL 学习信号。
- `Val gate`：验证集门槛。用固定验证集判断一个 adapter 是否超过基线；没过就不采用。
- `Test set`：测试集。只用于阶段性最终验收，不应频繁用来调参。

## 当前模型能做什么

当前模型已经不是候选框选择模型，而是 direct GUI action 模型。它能：

1. 读取浏览器截图和文本目标。
2. 根据历史动作和 verifier progress 判断下一步。
3. 输出合法 JSON action，例如 `click`、`type`、`press`、`scroll`、`finish`。
4. 使用 0-1000 归一化坐标点击页面元素。
5. 在 Playwright 浏览器环境中真实执行多步任务。
6. 完成本地 BrowserRL 中的 form、search、menu、table、todo、choice、advanced 等任务。

当前还不能称为通用桌面 agent，也不能保证泛化到任意真实网站或 OSWorld。它目前主要证明：Qwen2.5-VL-3B 在我们构建的可执行浏览器任务中已经具备多步 GUI 操作能力，并且 on-policy GRPO 能带来可测提升。

## 阶段 1：候选框路线与路线调整

最早我们做过 GUI candidate 路线：先生成候选框，再让模型从候选框中选择目标区域。这条路线工程上可控，但上限受候选池召回限制。如果目标框不在候选池中，模型无法选中。更重要的是，它容易把任务变成“选框排序”，而不是让 VLM 学会多步 GUI 操作。

因此我们把候选框、full-candidate score cache、candidate-only GRPO 降为辅助方向，主线转到可执行环境中的 direct GUI action。

## 阶段 2：本地 BrowserRL 环境

我们没有一开始就上 OSWorld，而是先用 Playwright 搭建本地浏览器环境。这样 reset、step、verifier 都更容易控制，适合快速验证算法闭环。

当前环境支持：

- 本地 HTML 任务页面。
- 每一步截图保存。
- 统一 JSON action schema。
- 0-1000 归一化坐标映射到真实浏览器坐标。
- verifier 自动判分。
- rollout 记录 task id、goal、action、reward、screenshot、execution status。

这一步的意义是：项目从静态截图数据转向可执行 agent 环境。

## 阶段 4：Form/Choice Repair 实验

我们尝试过对 form 和 choice 做定向 SFT 修复：

- repair adapter：`checkpoints/qwen25vl_3b_history_aware_sft_v3_form_choice_repair_v1_lora`
- best/root test100 success_rate：0.76
- v3 baseline test100 success_rate：0.82

这个实验虽然把 choice 从 0.6667 提到 0.8889，但 search 和 table 明显退化，所以没有采用 repair v1。结论是：只偏某些 family 的继续 SFT 容易造成能力干扰，后续应更多依赖 on-policy 数据、replay 或 KL 稳定项。

## 当前不足

1. 还没有跑 `test200`，所以不能称为最终模型。
2. advanced 从 0.7778 退到 0.6667，advanced_scroll/dialog 仍弱。
3. table 从 0.8800 退到 0.8400，需要 replay 或 KL 稳定。
4. 任务仍主要是本地 synthetic browser tasks，不是复杂真实网站或 OSWorld。
5. 当前是 Qwen2.5-VL-3B，不是 Qwen3-VL；Qwen2.5-VL-7B 已准备好，但还没复现实验。

## 下一步计划

优先级从高到低：

1. 用当前 RL adapter 作为 current model，扩大到 200-300 trainable groups。
2. 在 GRPO loss 中加入 v3 replay 或 KL 稳定项，避免旧能力退化。
3. 定向增加 advanced_scroll、advanced_dialog、choice_checkbox、table_action 的 on-policy 采样。
4. 继续用 `val_balanced_70` 做快速 gate，用 `val200` 做正式 gate。
5. 当 `val200 >= 0.82` 且主要 family 没有明显退化，再跑 `test200` 做阶段性验收。
6. 3B 闭环稳定后，把同样流程迁移到 Qwen2.5-VL-7B；之后再评估 Qwen3-VL。

## 阶段性判断

当前阶段的核心成果不是单纯把 success_rate 从 0.745 提到 0.790，而是 on-policy RL 闭环已经成立：当前模型能自己 rollout，环境能执行动作，verifier 能返回 reward，GRPO 能更新 LoRA adapter，val200 能验证是否真的提升。

接下来的关键不再是补静态截图数据，而是扩大 on-policy 样本、控制能力退化、让模型从自己的失败轨迹中学习。这才是 VLM agentic RL 的主线。
