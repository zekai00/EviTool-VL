# GUI Candidate RL 下一步算法改进报告

日期：2026-05-26  
项目路径：`/root/Workspace/VLM/EviTool-VL`

## 1. 当前结论

上一轮五步工作已经证明：

- GUI candidate RL 数据、SFT warmup、GRPO 训练、评估链路都能跑通。
- 但当前 `sequence-level GRPO` 没有带来可测提升。
- SFT warmup val100：`Pointing=19%`，`IoU@0.5=7%`。
- GRPO g2 100-step val100：`Pointing=18%`，`IoU@0.5=7%`。
- g4 有更好的组内探索，但 24GB 单卡 OOM；g2 能跑，但 `36/100` step 组内 reward 方差为 0，优势信号弱。

核心判断：**下一步不应该继续在同一配置上加 step，而应该改算法本身。**

当前方法把任务当成普通文本生成：

```text
Prompt + image -> generate '{"candidate_id":"cXX"}' -> parse -> reward
```

但这个任务的真实动作空间不是开放文本，而是当前截图里的有限候选集合：

```text
Action space = {c00, c01, ..., c59}
```

所以算法层面的改进应该是：**从自由文本 GRPO 改成候选约束的离散动作 GRPO。**

## 2. 推荐主方案：Candidate-Constrained GRPO

我建议下一步实现一个项目内定制算法，暂命名为：

```text
Candidate-Constrained GRPO, 简称 CC-GRPO
```

它和现在的 TRL GRPO 的关键区别如下：

| 维度 | 当前 GRPO | CC-GRPO |
|---|---|---|
| 动作 | 任意文本 completion | 当前样本中的合法 candidate id |
| 格式问题 | 需要 JSON parsing，可能无效 | 所有 action 天然合法 |
| reward | completion 解析后打分 | candidate bbox 直接打分 |
| 探索 | 采样文本，容易重复或跑偏 | 采样候选 id，探索集中在有效动作上 |
| 训练信号 | 经常 reward_std=0 | 通过 hard negatives 人为制造候选质量差异 |
| 算法重点 | 文本生成 RL | 离散候选选择 RL |

### 2.1 策略分布定义

对每个样本 `x`，有候选集合：

```text
C(x) = {c0, c1, ..., cN}
```

把每个 candidate id 映射成固定短 completion：

```json
{"candidate_id":"c00"}
{"candidate_id":"c01"}
...
```

模型不用自由生成，而是计算每个 candidate completion 的条件 log probability：

```text
s_theta(i | x) = log P_theta('{"candidate_id":"cXX"}' | prompt, image)
```

再在当前样本的候选集合内归一化：

```text
pi_theta(i | x) = softmax(s_theta(i | x) / tau)
```

这样模型仍然是原来的 Qwen2.5-VL LoRA，但训练目标从“生成任意文本”变成“在候选集合内给正确 candidate 更高概率”。

### 2.2 Reward v3：去掉格式奖励，保留候选质量奖励

CC-GRPO 中所有动作都是合法 candidate，所以不再需要 `format reward` 和 `valid reward`。reward 应只衡量候选本身对不对：

```text
raw_reward =
    0.35 * pointing
  + 0.30 * iou50
  + 0.20 * min(iou / 0.5, 1.0)
  + 0.15 * center_score
  - 0.05 * top1_wrong_penalty
```

含义：

- `pointing`：候选中心点是否落入 GT bbox。
- `iou50`：候选框与 GT bbox 的 IoU 是否大于等于 0.5。
- `iou shaped`：IoU 连续奖励，避免所有错误候选同分。
- `center_score`：中心距离越近越高，给 near miss 提供排序信号。
- `top1_wrong_penalty`：如果模型总选检测器 top1 但没命中，轻微惩罚。

注意：reward v3 的目标不是美化平均分，而是让候选之间产生稳定、可学习的相对差异。

### 2.3 GRPO advantage

对一个样本，从 `pi_theta` 中采样 `K` 个 candidate action：

```text
a1, a2, ..., aK ~ pi_theta(. | x)
```

计算每个 action 的 reward：

```text
r1, r2, ..., rK
```

再做组内标准化：

```text
A_i = (r_i - mean(r_group)) / (std(r_group) + eps)
```

这就是 GRPO 的核心：不训练 value model，只用同一 prompt 内多个动作的相对 reward 做 advantage。

### 2.4 Loss

建议先实现简化版 GRPO/PPO loss：

```text
ratio_i = exp(log_pi_theta(a_i | x) - log_pi_old(a_i | x))

L_policy = - mean(
    min(
        ratio_i * A_i,
        clip(ratio_i, 1 - eps, 1 + eps) * A_i
    )
)
```

同时加两个稳定项：

```text
L_kl = beta * KL(pi_theta(. | x), pi_ref(. | x))
L_entropy = - eta * H(pi_theta(. | x))

L_total = L_policy + L_kl + L_entropy
```

建议初始超参：

| 参数 | 建议值 |
|---|---:|
| `K` | 8 |
| `tau` | 1.2 |
| `clip_eps` | 0.2 |
| `beta` | 0.02 |
| `eta` | 0.01 |
| `lr` | 1e-6 到 2e-6 |
| `max_steps` | 100 debug，300 正式试跑 |

这样比当前 TRL GRPO 更适合 24GB 单卡，因为每个动作是短固定 candidate string，不需要采样很长 completion。

## 3. 关键算法改进点

### 3.1 动作空间约束

这是最重要的改动。

当前失败的一个原因是模型要同时学两件事：

1. 输出合法 JSON。
2. 选对 UI candidate。

CC-GRPO 直接把动作限制为合法 candidate id，让 RL 只优化第 2 件事。格式问题交给后处理或固定模板解决。

最终推理时可以仍然输出 JSON：

```json
{"candidate_id":"c12"}
```

但训练时不再让模型自由探索 JSON 字符串。

### 3.2 Hard-negative group 构建

当前 g2 GRPO 的问题是两个 sample 经常同分。解决办法不是简单加温度，而是主动构造更有区分度的候选组。

每个训练样本保留：

- `oracle candidate`：IoU/pointing 最好的正样本。
- `top-score wrong`：检测器排前但不命中的负样本。
- `near-center wrong`：中心离 GT 近但没命中的负样本。
- `high-IoU wrong`：IoU 接近但低于 0.5 的负样本。
- `same-text wrong`：OCR/text 或 label 相似但位置错误的负样本。
- `random wrong`：保持少量随机负样本，避免过拟合 hard negative。

这样每个 group 内天然有 reward 差异，GRPO 才有 advantage 信号。

建议新增脚本：

```text
scripts/build_gui_candidate_preference_data.py
```

输出字段建议：

```json
{
  "id": "...",
  "image": "...",
  "overlay_image": "...",
  "instruction": "...",
  "answer_bbox": [x1, y1, x2, y2],
  "candidates": [...],
  "candidate_rewards": {
    "c00": 0.12,
    "c01": 0.87
  },
  "positive_ids": ["c01"],
  "hard_negative_ids": ["c00", "c03", "c14"],
  "oracle_candidate_id": "c01",
  "oracle_rank": 2
}
```

### 3.3 Rank-balanced sampling

现在模型经常偏向 `c00/c01`，说明它利用了 candidate rank 先验，而不是理解 instruction。

训练集采样要平衡 oracle rank：

| 桶 | oracle rank |
|---|---|
| early | 1-5 |
| middle | 6-20 |
| late | 21-60 |

每个 batch 尽量混合三个桶。否则模型会学到“优先选前几个候选”的捷径。

### 3.4 Candidate coverage gate

如果候选集合里没有正确框，RL 不可能学会选对。

所以每次训练前必须先跑 coverage gate：

| 指标 | 当前 c60 val | 下一阶段目标 |
|---|---:|---:|
| Oracle hit | 62.88% | >= 75% |
| Oracle IoU@0.5 | 52.25% | >= 65% |
| Avg oracle IoU | 0.4426 | >= 0.55 |

上一轮 fused c80 比 c60 更差，说明不能简单把 OCR/OmniParser 混在一起后截断 top80。下一步 candidate 构建应改成：

1. 先生成大候选池，例如 200-300 个候选。
2. 对 OCR text box、text row、button rectangle、OmniParser icon 分 source 保留 quota。
3. 做 NMS/去重，避免同类候选挤占名额。
4. 再 query-aware rerank。
5. 最终截断到 60 或 80。

也就是从“全局排序后截断”改成“分源召回 + 去重 + query rerank”。

## 4. 推荐代码实现顺序

### Step A：新增候选级 reward 与 preference 数据

新增：

```text
rl/candidate_constrained.py
scripts/build_gui_candidate_preference_data.py
```

先不训练，只输出 preference 数据和 coverage/hard-negative 报告。

验收标准：

- 每条样本至少 1 个 positive。
- 每条样本至少 3 个 hard negatives。
- 训练集 `reward_std > 0` 的样本比例 >= 80%。
- oracle rank 三个桶都有足够样本。

### Step B：实现 candidate logprob scorer

新增函数：

```text
candidate_completion(candidate_id) -> str
compute_candidate_logprobs(model, processor, image, prompt, candidate_ids) -> Tensor[N]
```

注意事项：

- 对每个 candidate id 的固定 completion 计算 teacher-forced logprob。
- 使用 length-normalized logprob，避免字符串长度细微差异影响分数。
- candidate id 都是 `c00` 这种两位格式，减少 tokenization 差异。
- 先支持 batch 内一张图一个样本，稳定后再优化速度。

验收标准：

- 同一 prompt 下能输出每个 candidate 的 logprob。
- `softmax(logprobs)` 概率和为 1。
- oracle candidate 的初始概率不要求最高，但训练后应上升。

### Step C：先做 listwise supervised warmup

在 RL 前先做一个候选级 supervised warmup，比现在的 JSON SFT 更直接：

```text
L_listwise = - log pi_theta(oracle_candidate | x)
```

或者用 soft label：

```text
target_i = softmax(candidate_reward_i / target_tau)
L = CE(target, pi_theta)
```

建议先用 hard oracle CE，跑 1 epoch。

验收标准：

- oracle-hit val 上 `oracle candidate top1 rate` 明显提升。
- full val 的 Pointing / IoU@0.5 至少不下降。

### Step D：实现 CC-GRPO 100 step

新增：

```text
scripts/train_gui_candidate_cc_grpo.py
```

核心训练循环：

```text
for batch in dataloader:
    logp = compute_candidate_logprobs(...)
    pi = softmax(logp / tau)
    actions = sample K candidate ids from pi plus optional hard negatives
    rewards = reward_v3(actions)
    advantages = normalize(rewards within group)
    loss = clipped_policy_loss + KL + entropy
    backward/update LoRA
```

重要日志：

- `reward/mean`
- `reward/std`
- `frac_reward_zero_std`
- `action/oracle_sample_rate`
- `action/top1_bias_rate`
- `policy/entropy`
- `policy/kl_ref`
- `eval/pointing`
- `eval/iou50`

验收标准：

- `frac_reward_zero_std < 20%`
- `policy/entropy` 不快速塌缩
- oracle-hit val 有提升
- full val 不下降

### Step E：跑 300 step 正式试验

只有在 100 step 满足验收后才跑 300 step。

建议对照组：

| 实验 | 目的 |
|---|---|
| current SFT warmup | 当前基线 |
| sequence GRPO g2 | 已知弱 RL baseline |
| listwise warmup | 检查候选级监督收益 |
| CC-GRPO 100 | 验证新算法信号 |
| CC-GRPO 300 | 正式小规模 RL 结果 |

## 6. 预期收益与风险

### 预期收益

- 去掉 JSON 格式噪声。
- 单卡下可以使用更多候选动作样本。
- reward 方差由 hard negatives 保证，不再依赖自由文本采样碰运气。
- 更贴近任务本质：GUI grounding 是候选选择，不是开放文本生成。

### 主要风险

- candidate coverage 如果不提升，full val 上限仍会卡住。
- 计算所有 candidate logprob 会增加每 step 前向成本，需要先做小 batch。
- 如果只在 100 张截图上训练，模型可能记住截图布局。
- hard negatives 构造不好会让模型学偏，例如过度惩罚视觉相近但语义正确的候选。
