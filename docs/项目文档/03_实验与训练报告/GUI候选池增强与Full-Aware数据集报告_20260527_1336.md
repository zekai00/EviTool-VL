## 七步执行记录

1. 诊断 c60 baseline。
   - `val` oracle hit 为 62.88%，IoU@0.5 为 52.25%。
   - `table-cell`、`menu`、`tree-item`、`menu-item` 是主要瓶颈。

2. 构建第一版 local enhanced c100。
   - 输出：`outputs/gui_candidate_rl_os_atlas_linux_2k_enhanced_c100_local_20260527_1210`
   - 结果不好：`all` oracle hit 到 67.05%，但 IoU@0.5 降到 14.85%。
   - 判断：只做 quota/fusion 会引入大量大框，并丢失 c60 的紧框优势。

3. 修正为保留 `base_c60` 的 local enhanced c100。
   - 输出：`outputs/gui_candidate_rl_os_atlas_linux_2k_enhanced_c100_base_local_20260527_1210`
   - `all` oracle hit 75.00%，IoU@0.5 52.00%。
   - `val` oracle hit 78.72%，IoU@0.5 54.85%。

4. 做 DashScope VLM 补框 smoke test。
   - 使用模型：`gui-plus-2026-02-26`
   - 发现该模型返回 Qwen 0-1000 坐标，需要使用 `--coordinate-mode qwen1000`。
   - `qwen3.7-max` 当前 image request 格式返回 HTTP 400；`qwen3.6-plus` 家族 smoke 明显慢，本轮没有作为主增强模型。

5. 对 low-IoU 样本做 DashScope VLM 补候选。
   - 最终目录：`outputs/gui_candidate_rl_os_atlas_linux_2k_enhanced_c100_base_dashscope_20260527_1210`
   - `train` 调用 769 条，追加 965 个框，错误 0。
   - `val` 调用 191 条，追加 275 个框，错误 0。
   - 合计调用 960 条，追加 1240 个框，错误 0。

6. 重建 full-aware action 数据。
   - 输入：最终增强候选池。
   - 输出：`outputs/gui_candidate_rl_os_atlas_linux_2k_enhanced_c100_base_dashscope_full_aware_20260527_1329`
   - 训练过滤条件：`--require-train-oracle-hit --min-train-oracle-iou 0.05`
   - `train.jsonl` 为 1294 条，比旧 c60 full-aware 的 906 条更可用。

7. 做质量评测。
   - `python3 -m py_compile` 通过。
   - action id 全部存在于每行完整 `candidates` 中。
   - action reward 全部齐全。
   - `cc_action_ids` 与 `full_aware_action_ids` 一致。
   - zero reward std rate 为 0。
   - train/val image overlap 为 0。
   - `scripts/train_gui_candidate_cc_grpo.py --dry-run` 能加载最终 train 数据。

## 剩余瓶颈

`table-cell` 仍是最大召回瓶颈：

- final val: 21 条，oracle hit 4.76%，IoU@0.5 0.00%，avg IoU 0.0605
- final train: 84 条，oracle hit 5.95%，IoU@0.5 0.00%，avg IoU 0.0606

一些类型是“能点中但框不贴”：

- final val `menu`: oracle hit 90.00%，但 IoU@0.5 33.33%
- final val `check-menu-item`: oracle hit 100.00%，但 IoU@0.5 0.00%
- final val `menu-item`: oracle hit 100.00%，但 IoU@0.5 0.00%

这说明候选池已经更适合点击类 RL，但如果后续训练目标强依赖 IoU@0.5，还需要专门做 table/grid/menu tight box 规则。

## 建议下一步

1. 在最终 full-aware 数据上跑 current-model full-candidate score cache。
2. 用 `--model-score-jsonl` 重建 full-aware 数据，把 proxy false positives 替换为真实 current-model false positives。
3. 先做 listwise warmup，再做带 KL/reference logprobs 的 CC-GRPO。
4. 在 val100 或 val423 上做 full-candidates eval，对比：
   - old c60 SFT/GRPO
   - enhanced full-aware listwise
   - enhanced full-aware listwise + KL CC-GRPO

## 可复现实验命令

构建 local enhanced：

```bash
python3 scripts/build_gui_candidate_enhanced_data.py \
  --input-dir outputs/gui_candidate_rl_os_atlas_linux_2k_c60 \
  --output-dir outputs/gui_candidate_rl_os_atlas_linux_2k_enhanced_c100_base_local_20260527_1210 \
  --max-candidates 100 \
  --pool-size 240 \
  --min-area 20 \
  --ocr-engine easyocr \
  --include-ocr \
  --include-omniparser \
  --include-input-candidates \
  --seed 42
```

DashScope VLM 补框：

```bash
python3 scripts/augment_gui_candidates_dashscope.py \
  --input-dir outputs/gui_candidate_rl_os_atlas_linux_2k_enhanced_c100_base_local_20260527_1210 \
  --output-dir outputs/gui_candidate_rl_os_atlas_linux_2k_enhanced_c100_base_dashscope_20260527_1210 \
  --splits train \
  --augment-iou-below 0.5 \
  --max-vlm-boxes 5 \
  --max-candidates 110 \
  --coordinate-mode qwen1000 \
  --models gui-plus-2026-02-26 \
  --sleep-sec 0.05
```

重建 full-aware：

```bash
python3 scripts/build_gui_candidate_full_aware_data.py \
  --input-dir outputs/gui_candidate_rl_os_atlas_linux_2k_enhanced_c100_base_dashscope_20260527_1210 \
  --output-dir outputs/gui_candidate_rl_os_atlas_linux_2k_enhanced_c100_base_dashscope_full_aware_20260527_1329 \
  --report reports/gui_candidate_full_aware_dataset_enhanced_c100_base_dashscope_20260527_1329.md \
  --max-actions 16 \
  --min-train-oracle-iou 0.05 \
  --require-train-oracle-hit \
  --seed 42
```
