• 这些指标分成三类：任务正确率、tool-use 协议质量、工具执行质量。代码位置主要在 EviTool-VL/eval/eval_baseline.py:144 和 EviTool-VL/eval/
  eval_tool_baseline.py:371。

  指标含义
  text_exact_match
  用于 ChartQA / DocVQA / AI2D。先把预测和答案做简单归一化：小写、去多余空格、去 answer: 前缀、去末尾标点，然后完全相等才算对。AI2D 会额外接受选项字
  母，比如 B，或选项文本。

  text_relaxed_match
  文本题的宽松匹配。只要 exact match 对，或者数值题第一处数字与答案第一处数字的相对误差小于等于 3%，就算对。AI2D 的 relaxed 和 exact 基本一致。

  grounding_iou_50
  用于 ScreenSpot GUI grounding。预测里提取前 4 个数字作为 bbox，计算预测框和真值框的 IoU：

  IoU = intersection_area / union_area

  IoU >= 0.5 算成功。

  grounding_pointing_accuracy
  也是 GUI grounding。若模型输出 bbox，就取预测 bbox 的中心点；若只输出点，就直接用该点。只要这个点落在 GT bbox 里，就算成功。这个比 IoU@0.5 宽松，更
  接近“点中 UI 元素即可点击”的实际需求。

  bbox_parse_rate
  GUI 题里能不能从模型输出解析出 bbox 或 point。它只检查格式，不代表位置正确。

  mean_iou
  GUI 题所有样本的平均 IoU。比 IoU@0.5 更连续，但会被大框、偏移框影响。

  empty_output_rate
  预测为空的比例。

  avg_latency_sec
  单样本平均耗时。这里包含模型生成和工具调用耗时，OCR 会明显拉高。

  final_parse_rate
  tool-use 模式下，最终输出能否解析成合法 FINAL JSON，例如：

  {"reasoning":[{"step":"...","evidence":["ev_001"]}],"answer":"..."}

  final_recovery_rate
  最终 JSON 不合法，但脚本用正则从原始输出里救出了 answer。这类样本可以参与答案打分，但不算严格协议成功。

  samples_with_parse_error_rate
  循环中任意一步出现 JSON 解析错误的样本比例。

  samples_with_protocol_error_rate
  违反我们定义的工具协议的样本比例，比如第一步直接 final、调用工具后 final 没引用 evidence id 等。当前没有开启强制重试，所以有些违规样本仍会继续打
  分。

  avg_tool_calls / total_tool_calls / max_tool_calls
  每个样本平均、总计、最大工具调用次数。

  tool_success_rate
  所有工具 observation 里 ok=True 的比例。注意它只代表工具执行成功，不代表工具结果对答案有帮助。

  tool_counts
  模型请求过的工具名字计数。这里会保留非法工具名，所以你会看到 count、final 这种值，它们是诊断信号。

  evidence_reference_rate
  final reasoning 里是否引用了至少一个 evidence id。

  evidence_closed_rate
  final 引用的 evidence id 是否都来自真实 observation。它只检查“引用闭环”，不检查 evidence 是否真的支持答案。

  这样选合理吗
  作为当前阶段的 first-pass eval，是合理的。它把三件事拆开了：答案是否正确、模型是否遵守 action-observation-final 协议、工具是否正常执行。这样能定位
  问题来源，而不是只看一个总分。

  但它还不是最终论文级或生产级评测。主要限制是：

  - 文本匹配偏简单，数值只取第一处数字，复杂单位、多数字答案、同义表达处理不充分。
  - ChartQA / DocVQA / AI2D 后面应接官方评测脚本或更接近官方口径的指标。
  - ScreenSpot 的 pointing_accuracy 对 GUI 点击很合理，IoU@0.5 对小 UI 元素偏严格，但能衡量 bbox 质量。两个都保留是对的。
  - evidence_closed_rate 只能证明引用 id 合法，不能证明推理真的用了 evidence。
  - eval_mini_100 太小，只适合快速迭代；后面要上 full eval，并加 bootstrap 置信区间。
  - tool_success_rate 不等于工具有用率，后面要加 tool recall / answer-in-observation 等工具有效性指标。

  下一步重点

  1. 先提升协议稳定性
     Qwen2.5-VL-3B 的 protocol error 是 73%，这是最先要解决的。做法是构造 SFT 轨迹，让模型稳定学会：

  assistant action JSON -> tool observation -> assistant final JSON

  训练时只对 assistant 输出算 loss，observation 作为上下文，不训练模型复述 observation。

  2. 提升 GUI grounding
     这是目前最大业务短板。direct baseline 的 ScreenSpot IoU@0.5 是 0%，tool-use 后也只有 12%。下一步应该增强 detect/ui + click：

  - 把 OCR text boxes、UI contour boxes、图标候选框合并成 candidate list。
  - 给每个候选框返回 candidate_id, bbox, center, text, visual_type。
  - final 阶段优先输出候选框或 click center。
  - 增加候选召回指标：GT 是否落在 top-k candidates 中。
  - 用 GT bbox 合成 oracle traces，训练模型选择正确 candidate。

  3. 提升工具选择质量
     Qwen3 协议很好，但 grounding 仍差，说明它会用工具但不会总是选对工具或读对 observation。需要做工具轨迹数据：

  - ChartQA：detect(bar) / ocr(axis,label) / measure。
  - DocVQA：ocr 优先，必要时 zoom 到区域。
  - GUI：detect(ui) -> click(candidate)。
  - AI2D：inspect/crop/zoom 后 final。

  4. 加强非法工具和 schema 控制
     当前模型会偶尔输出 count、final 作为工具名。需要增加：

  - unknown_tool_rate
  - tool_first_rate
  - valid_action_schema_rate
  - valid_final_schema_rate

  并在训练数据中加入非法工具负例，或者在推理时加 schema retry。

  5. 再做 LoRA SFT，而不是继续调 prompt
     当前结果已经说明 prompt-only 到顶很快。建议顺序：
  6. 用现有 tools 生成 1k-5k 条 action-observation-final 训练轨迹。
  7. 先训 Qwen2.5-VL-3B LoRA，目标是协议稳定。
  8. 再训 Qwen3-VL-4B LoRA，目标是工具选择和 grounding。
  9. 每轮都跑 eval_mini_100，然后再跑 full eval。
  10. grounding 单独看 pointing_accuracy、IoU@0.5、candidate recall@k。

  短期最有价值的改进是：生成 GUI grounding oracle tool traces，然后 SFT。因为现在工具闭环已经通了，最大收益会来自让模型学会“看候选框、选候选框、引用
  evidence、输出稳定 bbox”。
