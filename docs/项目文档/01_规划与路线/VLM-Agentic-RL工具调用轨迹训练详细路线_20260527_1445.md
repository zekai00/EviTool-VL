# VLM Agentic RL 工具调用轨迹训练详细路线

- 生成时间：2026-05-27 14:45 CST
- 项目根目录：`/root/Workspace/VLM`
- 当前仓库：`/root/Workspace/VLM/EviTool-VL`
- 当前算力：2 x NVIDIA GeForce RTX 3090, each 24576 MiB
- 本地可用底模：`Qwen2.5-VL-3B-Instruct`、`Qwen3-VL-4B-Instruct`、`Qwen3-VL-8B-Instruct`
- 结论：可以做 tool-call trajectory RL；下一阶段应从 candidate RL 转向“工具调用轨迹训练 + evidence-closed agentic RL”。

## 1. 核心判断

Qwen2.5-VL/Qwen3-VL 都可以作为 tool-call trajectory RL 的底模，但第一阶段不应直接上长轨迹在线 RL。

更稳的顺序是：

1. 先用现有 GUI/candidate/tool 基础设施生成 1k-2k 条可审计工具调用轨迹。
2. 对 `Qwen2.5-VL-3B-Instruct` 做 LoRA SFT，让模型先学会合法工具调用格式、读取 observation、引用 evidence、终止输出。
3. 同步小规模验证 `Qwen3-VL-4B-Instruct` 的训练栈；如果 loader、processor、TRL/Unsloth 支持稳定，再把 4B 作为主力。
4. `Qwen3-VL-8B-Instruct` 先作为 teacher、silver trace generator、评测模型或高质量 SFT 对照，不建议作为第一轮在线 RL 主力。

原因：

- Qwen2.5-VL 官方技术报告明确将其定位为可进行 reasoning、tool usage、computer/mobile task execution 的 interactive visual agent。
- Qwen2.5-VL 官方博客强调其支持 bbox/point grounding 和结构化 JSON 输出，这正好对应 GUI grounding 和 action schema。
- Qwen3-VL 模型卡强调 Visual Agent 能操作 PC/mobile GUI、识别元素、调用工具、完成任务，并且有更强空间感知和 2D grounding。
- TRL 当前已支持 GRPO agent training 的 `tools` 和 `environment_factory`，并支持工具返回多模态 observation；这说明从工程接口上已经能训练工具调用 agent。

所以问题不是“Qwen 能不能做”，而是“如何在 2x3090 上把问题拆成可训练、可验证、不会 reward hacking 的小环境”。

## 2. 底模选择

### 推荐主线

第一优先级：`Qwen2.5-VL-3B-Instruct`

- 优点：已有本地模型、已有训练配置、已有 GUI candidate RL 经验，显存压力最低。
- 论文和社区已有 GUI-RL 例子使用 `Qwen2.5-VL-3B-Instruct` 做 GUI grounding RL；GUI-G1 也是基于 3B 做 R1-like GUI grounding 训练。
- 适合先跑通 trajectory SFT、reward、rollout、evaluation 的完整闭环。

第二优先级：`Qwen3-VL-4B-Instruct`

- 优点：更贴近下一代 visual agent 能力，本地已有模型，模型卡明确强调 GUI agent/tool 能力。
- 风险：训练栈可能比 Qwen2.5-VL 更容易遇到 processor/chat-template/TRL 版本问题。
- 建议：不要等 4B 完全稳定才开始，先用 3B 跑通闭环；4B 作为并行迁移目标。

第三优先级：`Qwen3-VL-8B-Instruct`

- 优点：能力更强。
- 风险：2x3090 上在线 RL rollout 成本高，image token + multi-turn + group generation 会放大显存和时间压力。
- 建议用途：teacher trace generation、DashScope 结果校验、本地强 baseline、少量 LoRA SFT 对照。

### 是否要换底模

暂时不建议换到 InternVL/LLaVA/其他 VLM。当前项目已有 Qwen 系工具链、数据格式、训练配置和候选池，迁移底模会消耗大量工程时间，且不一定提升 GUI tool-call trajectory 的核心问题。

只有在下面情况出现时才考虑换底模：

- Qwen3-VL 训练栈持续无法稳定加载或训练。
- Qwen2.5-VL-3B 在工具调用格式上经过 SFT 后仍严重不服从 JSON protocol。
- 发现某个底模已经有成熟、可复现的 VLM agentic RL repo，且支持本地 24GB GPU。

## 3. 现有方法与对本项目的启发

### Agent Lightning

Agent Lightning 将 agent 执行过程建模为 MDP，并提出 hierarchical RL / credit assignment，把复杂 agent trajectory 拆成可训练 transition。这对本项目很重要，因为我们的 trace 不是单次输出，而是：

```text
screenshot + instruction
-> tool call
-> observation
-> tool call
-> observation
-> final bbox/click/answer
```

启发：不要把整条轨迹简单拼接后只给一个最终 reward；要把每一次工具调用、observation 使用、最终答案拆成 transition，并给可解释的局部 credit。

### RAGEN / StarPO

RAGEN 指出 multi-turn agent RL 容易出现 reward variance cliff、gradient spike、浅层策略和幻觉式 reasoning；没有细粒度 reward 时，agent reasoning 很难自然涌现。

启发：

- 不能只给 final answer reward。
- 要有 tool validity、evidence closure、observation usefulness、format reward。
- 初期 horizon 要短，逐步从 1-2 步工具扩到 3-4 步。

### AgentGym-RL

AgentGym-RL 强调 modular/decoupled environment，以及从 exploitation 到 exploration 的交互长度 curriculum。

启发：

- 先限制工具步数，让模型学会短链路。
- 再逐步放开 max tool steps。
- 环境和训练器解耦，方便先跑 scripted oracle，再跑 model rollout。

### Pixel Reasoner / DeepEyes

Pixel Reasoner 和 DeepEyes 的共同点是：不要让 VLM 只在文本空间“想”，而要让模型主动执行 zoom/crop/select-frame 等视觉操作，把视觉证据带回 reasoning。

启发：

- 本项目的工具不应只是 debug 辅助，而应成为 policy action。
- `crop`、`zoom`、`ocr`、`detect` 是像素空间探索动作。
- evidence-closed reward 要检查模型是否真的使用了工具 observation，而不是凭先验猜答案。

### GUI-R1 / GUI-G1

GUI-R1 用统一 action space 和 GRPO 提升 GUI agent 执行能力；GUI-G1 进一步指出 GUI grounding RL 里常见 reward hacking：模型可能通过输出过大的 box 获取 hit reward，所以要加 box size constraint，并且要做 difficulty-aware optimization。

启发：

- GUI grounding reward 不能只有 hit/pointing。
- 需要 IoU、box area、候选召回、困难样本权重。
- 长 CoT 未必提升 GUI grounding，工具调用 trace 中的 `thought` 应短而可执行，不应训练冗长推理。

## 4. 数学建模

### 4.1 任务定义

每个样本：

```text
x = (I, u, y*)
```

其中：

- `I` 是 GUI 截图或视觉输入。
- `u` 是用户 instruction/question。
- `y*` 是隐藏 ground truth，例如目标 bbox、click point、文本答案。

agent 在第 `t` 步看到的历史：

```text
h_t = (I, u, a_1, o_1, ..., a_{t-1}, o_{t-1})
```

action space：

```text
A = A_tool ∪ A_final
```

工具动作：

```text
A_tool = {
  inspect(),
  detect(mode, query, region),
  ocr(region),
  crop(bbox),
  zoom(bbox, scale),
  measure(...),
  select_candidate(candidate_id),
  click(point | bbox | candidate_id)
}
```

最终动作：

```text
A_final = {
  answer_bbox([x1,y1,x2,y2]),
  answer_text(text),
  finish(answer, evidence_ids)
}
```

工具环境：

```text
o_t = T(a_t; I, h_t)
```

策略模型：

```text
a_t ~ π_θ(a_t | h_t)
```

轨迹：

```text
τ = (h_1, a_1, o_1, ..., h_T, a_T)
```

目标：

```text
max_θ J(θ) = E_{x~D, τ~π_θ,T} [ R(τ; x) ] - β KL(π_θ || π_ref)
```

其中 `π_ref` 是 SFT 后的 reference model，`β` 控制 KL。

### 4.2 Reward 分解

建议总 reward：

```text
R(τ; x) =
  w_ans   r_ans
+ w_ev    r_evidence
+ w_fmt   r_format
+ w_tool  r_tool
+ w_hard  r_hard
- w_cost  c_tool
- w_area  p_area
- w_bad   p_bad
```

GUI grounding 的最终答案 reward：

```text
r_ans = 0.7 · 1[center(ŷ) ∈ y*] + 0.3 · min(IoU(ŷ, y*) / 0.5, 1)
```

其中 `ŷ` 是模型最终输出 bbox。如果最终选择 candidate：

```text
ŷ = bbox(candidate_id)
```

Evidence closure reward：

```text
r_evidence = 1[final cites valid evidence] · (0.5 + 0.5 · E(τ))
```

证据有效性：

```text
E(τ) = max_{t,b∈obs_t} g(IoU(b, y*)) · m(b, u, obs_t)
```

其中：

- `g(IoU)=min(IoU/0.5,1)`。
- `m(b,u,obs_t)` 表示证据是否与 instruction 匹配，可由 OCR 文本命中、UI label 命中、candidate role 命中或 teacher verifier 判断。

工具格式 reward：

```text
r_format =
  0.4 · 1[valid JSON]
+ 0.3 · 1[valid action schema]
+ 0.3 · 1[valid final schema]
```

工具执行 reward：

```text
r_tool = (# successful tool calls) / max(1, # tool calls)
```

工具成本：

```text
c_tool = min(c_max, λ_step · max(0, N_tool - N_free))
```

box area penalty：

```text
p_area = 1[ area(ŷ) / area(I) > ρ_max ] + 1[ area(ŷ) < ρ_min ]
```

用于缓解“输出超大框也能 hit”的 reward hacking。

### 4.3 GRPO 训练公式

对同一个样本 `x` 采样 `G` 条 rollout：

```text
{τ_i}_{i=1}^G ~ π_{θ_old}
```

每条轨迹得到 reward：

```text
R_i = R(τ_i; x)
```

组内归一化 advantage：

```text
A_i = (R_i - mean(R_1,...,R_G)) / (std(R_1,...,R_G) + ε)
```

token/action 级概率比：

```text
ρ_{i,t} = π_θ(a_{i,t}|h_{i,t}) / π_{θ_old}(a_{i,t}|h_{i,t})
```

GRPO/PPO-style clipped objective：

```text
L_GRPO(θ) =
- 1/G · Σ_i Σ_t min(
    ρ_{i,t} A_i,
    clip(ρ_{i,t}, 1-ε_clip, 1+ε_clip) A_i
  )
+ β · KL(π_θ || π_ref)
```

对本项目，初期可把 reward 放在 trajectory level；第二阶段再引入 step-level credit。

### 4.4 分层 credit assignment

将轨迹拆成两层：

```text
z_t ∈ {tool, final}
a_t ~ π_θ(a_t | h_t, z_t)
```

工具调用 credit：

```text
A_t^tool = r_t^tool + γ V(h_{t+1}) - V(h_t)
```

最终答案 credit：

```text
A_T^final = r_ans + r_evidence - V(h_T)
```

早期不用训练独立 critic，也可以用可解释的 counterfactual credit：

```text
Δ_t = S(final | h_T) - S(final | h_T \ o_t)
```

其中 `h_T \ o_t` 表示把第 `t` 个 observation mask 掉后重新打分。若 `Δ_t` 大，说明该 observation 对最终答案有贡献；若 `Δ_t≈0`，说明工具调用可能是无效装饰。

## 5. 可做的算法创新

### 5.1 Evidence-Closed Hierarchical GRPO

核心思想：把“最终答对”和“证据闭环”同时优化，避免模型只学会输出正确 bbox，而没有真正调用和使用工具。

改造点：

- trajectory reward 中显式加入 `r_evidence`。
- final JSON 必须引用 evidence ids。
- 被引用 evidence 要能定位到目标区域或目标文本。
- 对工具 observation 做 counterfactual credit，给有效工具调用更高 advantage。

论文表达可以是：

> We optimize VLM agents with evidence-closed hierarchical GRPO, where the final answer is rewarded only when it is both task-correct and supported by valid visual-tool observations.

### 5.2 Candidate-as-Option 混合动作空间

当前 candidate RL 不丢弃，而是升级成一个 terminal option：

```text
a_t = select_candidate(candidate_id)
```

模型可以走两条路线：

```text
detect/ocr/crop/zoom -> select_candidate -> finish
detect/ocr/crop/zoom -> answer_bbox -> finish
```

这样既保留候选框召回的效率，又允许候选池缺失时直接输出 bbox。

混合策略：

```text
π(a|h) = π_z(z|h) · π(a|h,z)
```

其中：

- `z=candidate` 时，从候选 id 中选。
- `z=free_bbox` 时，直接生成 bbox。
- `z=tool` 时，调用视觉工具。

### 5.3 Difficulty-Aware Group Scaling

GUI-G1 的经验说明，简单在线 RL 容易过拟合 easy samples。本项目可以根据候选池 oracle 质量定义难度：

```text
d(x) = 1 - max_c IoU(b_c, y*)
```

或者：

```text
d(x) = 1[oracle_hit=0] + 0.5·1[max_iou<0.5] + entropy(candidate_scores)
```

advantage scaling：

```text
A_i' = (1 + α d(x)) · A_i
```

这样 hard GUI 样本不会在 group normalization 中被 easy samples 淹没。

### 5.4 Tool-Use Curiosity Reward

借鉴 Pixel Reasoner 的 curiosity-driven idea，但约束在 evidence usefulness 上：

```text
r_curiosity(t) = H(p_{t-1}) - H(p_t)
```

其中 `p_t` 是模型或 verifier 在看到第 `t` 个 observation 后对正确 candidate/bbox 的分布。不鼓励乱调用工具，只奖励能降低不确定性的 observation。

```json

  }

```

这类数据既能做 SFT，也能做 offline replay evaluation，还能作为 GRPO rollout 的 prompt seed。

### 6.2 轨迹模板

第一版 1k-2k 条数据不要追求复杂，应覆盖几类稳定路径：

1. `detect_ui -> select_candidate`
   - 适合 button/menu/menu-item/check-menu-item。
   - 从 refined candidate pool 选择 oracle candidate。

2. `ocr_full -> select_candidate`
   - 适合文本可见、目标 instruction 包含可 OCR 文本的样本。
   - evidence 来自 OCR span 和 candidate bbox overlap。

3. `crop_region -> ocr_region -> select_candidate`
   - 适合 menu、toolbar、table 局部密集区域。
   - crop region 可来自 oracle candidate 周围扩张框，或来自 layout detector。

4. `detect_layout/table -> answer_bbox`
   - 适合 table-cell 候选容易缺失或需要结构推断的样本。
   - 允许 free bbox，不强制 candidate_id。

5. `detect_ui -> reject_bad_candidate -> answer_bbox`
   - 用于训练候选缺失时不要硬选错误 candidate。
   - final action 使用 `answer_bbox`。

### 6.3 质量过滤

构建后必须保留这些过滤指标：

- JSON schema valid rate。
- tool execution success rate。
- final parseable rate。
- evidence id valid rate。
- evidence-target IoU。
- final bbox IoU / pointing。
- candidate oracle hit。
- train/val image overlap。
- 每类 data_type 的样本数和指标。

建议第一版验收线：

- schema valid rate >= 99%
- tool success rate >= 95%
- evidence id valid rate >= 95%
- GUI final IoU@0.5 >= 75% for oracle/silver trajectories
- train/val image overlap = 0

目标：先证明数据格式、工具执行、reward 和 SFT loader 全部能跑。

输出：

- `outputs/agentic_traces/gui_tool_traces_smoke_500_YYYYMMDD_HHMM/train.jsonl`

### 阶段 D：构建 2k full traces

目标：覆盖 refined full-aware train/val。

输出：

- `outputs/agentic_traces/gui_tool_traces_2k_YYYYMMDD_HHMM/train.jsonl`
- `outputs/agentic_traces/gui_tool_traces_2k_YYYYMMDD_HHMM/val.jsonl`
- `reports/gui_tool_traces_2k_quality_YYYYMMDD_HHMM.md`

可使用 DashScope/Qwen3.7-max 的位置：

- 生成轨迹模板选择：给定 screenshot metadata、instruction、candidate list，让 teacher 判断应走 `detect`、`ocr`、`crop` 还是 `answer_bbox`。
- 验证 evidence 是否支持 final answer。
- 对 ambiguous GUI label 做辅助判断。

不建议让 API 直接生成最终 bbox 后无校验写入训练集。最终 bbox 必须通过本地 GT/IoU/candidate oracle 校验。

### 阶段 E：Trajectory SFT

第一轮：

```text
Qwen2.5-VL-3B-Instruct + LoRA
```

训练目标：

- 学会 JSON action/final protocol。
- 学会短工具链：1-2 步。
- 学会引用 evidence id。
- 学会在候选存在时 `select_candidate`，候选缺失时 `answer_bbox`。

第二轮：

```text
Qwen3-VL-4B-Instruct + LoRA
```

与 3B 做同样数据、同样评测，对比：

- action schema valid rate
- tool call success rate
- evidence closure
- final IoU/pointing
- direct/candidate/tool-agent 三模式差异

### 阶段 F：Agentic GRPO / RLVR

初期设置：

- group size `G=4`，稳定后到 `G=8`。
- max tool steps 从 1 开始，逐步到 2/3。
- LoRA/QLoRA。
- reference model 使用 SFT checkpoint。
- reward 使用第 4 节分解版本。

训练策略：

1. short-horizon GRPO：只允许一次工具调用后 final。
2. two-step GRPO：允许 `detect/ocr -> crop/zoom/measure -> final`。
3. candidate-free mixture：部分样本隐藏 candidates，迫使 `answer_bbox`。
4. hard-sample upweight：对低 oracle IoU、menu/table、密集 UI 加 advantage scaling。

### 阶段 G：评测矩阵

每次训练后必须同时评测三种模式：

1. Direct mode
   - 不给工具，不给候选，直接输出 bbox/answer。

2. Candidate mode
   - 给候选列表，只选 candidate。
   - 对比当前 candidate RL。

3. Tool-agent mode
   - 模型自己调用工具。
   - 评估 tool validity、evidence closure、final answer。

核心指标：

- final IoU@0.5 / pointing accuracy
- oracle upper bound gap
- tool call valid rate
- evidence-closed rate
- average tool calls
- hard split accuracy
- menu/table/check-menu/menu-item 分项
- train/val leakage check

## 8. 是否需要新开项目

不建议现在新开项目。

理由：

- 当前 repo 已经有工具系统、candidate pool、reward、rollout env、eval scripts。
- agentic RL 和 candidate RL 是同一条 evidence-closed visual tool agent 主线的上下游，不是完全不同项目。
- 新开项目会复制工具、数据路径、评测逻辑，增加维护成本。

建议做法：

```text
EviTool-VL/
  tools/
  rl/
  agentic_rl/
  scripts/build_gui_tool_trajectories.py
  scripts/eval_tool_trajectories.py
  configs/sft_gui_tool_trajectory_qwen25vl_3b_lora.yaml
  configs/grpo_gui_tool_agent_qwen25vl_3b_lora.yaml
```

只有当准备对外发布一个干净 repo，或者 agentic RL 代码已经明显挤压原有结构时，再拆成新项目。

## 9. 最近一周执行计划

第一天：

- 定义 trajectory JSON schema。
- 写 `build_gui_tool_trajectories.py` smoke 版。
- 从 refined full-aware 中抽 500 条，生成可执行 traces。

第二天：

- 实现 trajectory quality evaluator。
- 输出 smoke 数据质量报告。
- 修 schema、evidence id、tool observation compact 逻辑。

第三天：

- 构建 2k full traces。
- 按 data_type 输出质量表。
- 对低质量样本做过滤或 teacher verifier。

第四天：

- 跑 `Qwen2.5-VL-3B-Instruct` trajectory SFT。
- 评测 direct/candidate/tool-agent 三模式。

第五天：

- 小规模迁移 `Qwen3-VL-4B-Instruct`。
- 对比 3B/4B 的格式服从和工具使用。

第六天：

- 接入 GRPO environment_factory 或现有 rollout env。
- 跑 100-300 step short-horizon GRPO smoke。

第七天：

- 输出训练报告。
- 决定是否扩大到 5k+ traces、是否把 DashScope teacher 引入 verifier、是否启动 long-horizon RL。

## 10. 一句话结论

当前 GUI candidate RL 已经足够作为底层 grounding option；下一步最有意义的主线不是继续手工增强候选框，而是构建可审计 tool-call trajectories，先 SFT 再做 evidence-closed hierarchical GRPO，让 Qwen2.5-VL/Qwen3-VL 学会主动调用工具、读取 observation、引用视觉证据并完成 GUI grounding。
