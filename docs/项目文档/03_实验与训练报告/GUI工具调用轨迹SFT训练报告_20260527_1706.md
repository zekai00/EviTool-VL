## 5. 关键发现

本次 SFT 成功学到了部分工具调用表层协议：

- 能生成 JSON。
- 能生成 `detect` action。
- 在标准历史下能稳定生成 `click` action schema。

但它没有学会可靠的候选选择策略：

- teacher-forced 条件下，模型倾向选择固定高排位 bbox。
- 这与此前 candidate RL 中的 top-rank/c00 shortcut 是同一类问题。
- 单纯 trajectory SFT，尤其只用成功轨迹，很容易让模型模仿“选一个看起来常见的候选”，而不是学会从 50 个候选中辨别目标。

因此，该 adapter 当前适合作为 protocol warmup，不适合作为直接进入 agentic RL 的最终初始策略。

## 6. 结论

训练已完成，但质量结论不是“可以直接 RL”，而是：

> Qwen2.5-VL-3B 已经能被 SFT 拉到工具协议附近，但候选选择能力仍然明显不足。下一步需要构建更强的 candidate-selection trajectory SFT 或把已有 candidate policy 接入 agent，而不能只依赖这版 clean-candidate SFT。

## 7. 下一步建议

优先做三件事：

1. 构建 `trajectory_sft_v2`。
   - action 从 `click(bbox)` 改成 `select_candidate(candidate_id, bbox)`。
   - observation 使用 8-16 个 action candidates，而不是 50 个全量候选。
   - candidate 顺序做随机化或 hard-negative 重排，降低 top-rank shortcut。

2. 混合已有 candidate SFT 数据。
   - 把“候选选择”作为一个明确子任务继续训练。
   - trajectory SFT 负责工具协议，candidate SFT 负责选择能力。

3. 再做 current-model rollout。
   - 评估 action JSON strict rate。
   - 评估 click/select_candidate IoU。
   - 再决定是否进入 GRPO。
